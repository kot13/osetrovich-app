# Osetrovich Constitution

Мобильное приложение интернет-магазина [osetrovich.ru](https://osetrovich.ru). Конституция
определяет неизменяемые принципы разработки, технические ограничения и порядок внесения
изменений.

## Core Principles

### I. Flutter — кроссплатформенная разработка

Приложение MUST разрабатываться на **Flutter** с поддержкой **Android** и **iOS**. Общая
бизнес-логика и UI MUST реализовываться в единой кодовой базе; платформенно-специфичный
код допускается только при объективной необходимости (нативные SDK, разрешения, store
integration).

**Обоснование**: единая кодовая база снижает стоимость сопровождения и обеспечивает
паритет функциональности на обеих платформах.

### II. Русскоязычный интерфейс

Весь пользовательский интерфейс приложения MUST быть **только на русском языке**: тексты
кнопок, заголовков, сообщений об ошибках, пустых состояний, подсказок и уведомлений.
Локализация на другие языки вне scope, если не принято отдельное решение через поправку к
конституции.

**Обоснование**: целевая аудитория магазина osetrovich.ru — русскоязычные пользователи.

### III. Тестирование основного функционала (ОБЯЗАТЕЛЬНО)

Для **всего основного функционала** MUST писаться автоматизированные тесты до или
одновременно с реализацией. Минимальный охват:

- **Unit-тесты** — бизнес-логика, модели, маппинг данных, use cases.
- **Widget-тесты** — ключевые экраны и интерактивные компоненты.
- **Integration-тесты** — сквозные пользовательские сценарии (user stories из spec.md).

Вспомогательный или тривиальный код (генераторы, конфигурация без логики) MAY обходиться
без тестов; решение MUST быть явно задокументировано в plan.md.

**Обоснование**: магазинное приложение критично к корректности заказов, каталога и
платёжных сценариев; регрессии недопустимы.

### IV. Flutter Best Practices

Код MUST соответствовать общепринятым best practices экосистемы Flutter/Dart:

- `flutter analyze` без ошибок; форматирование через `dart format`.
- Чёткое разделение слоёв: presentation / domain / data.
- Управление состоянием — предсказуемое и тестируемое (без бизнес-логики в виджетах).
- Именование и структура каталогов по feature-first или согласованной схеме из plan.md.
- Асинхронные операции с корректной обработкой ошибок и состояний загрузки.
- Доступность и адаптивность layout под типовые размеры экранов iOS и Android.

**Обоснование**: единые стандарты упрощают ревью, онбординг и долгосрочную поддержку.

### V. Tab Bar — основная навигация

Главная навигация приложения MUST строиться на **нижнем Tab Bar** (BottomNavigationBar или
эквивалент). Вкладки MUST отражать ключевые разделы магазина (например: каталог, корзина,
профиль — точный состав определяется в spec.md фичи).

Вложенная навигация (стек внутри вкладки, модальные окна) MAY использоваться для детальных
экранов, но корневой паттерн перемещения между разделами — Tab Bar.

**Обоснование**: единообразная навигация соответствует ожиданиям пользователей
e-commerce и снижает когнитивную нагрузку.

### VI. OpenAPI-контракт и моки

Бэкенд — **REST API**. Спецификация API MUST описываться в проекте в формате **OpenAPI**
(каталог `openapi/` или путь из plan.md). До подключения боевого API приложение MUST
использовать **моки**, полностью соответствующие контракту OpenAPI (схемы, коды ответов,
структура ошибок).

Любое изменение контракта MUST начинаться с обновления OpenAPI, затем моков, затем
клиентского кода. Расхождение моков и спецификации MUST считаться дефектом.

**Обоснование**: contract-first подход позволяет параллельно развивать клиент и бэкенд без
блокировок.

### VII. Спецификации на русском языке

Все артефакты Spec Kit (`spec.md`, `plan.md`, `tasks.md`, `research.md`, чеклисты и
сопутствующая проектная документация в `specs/`) MUST писаться **только на русском
языке**. Идентификаторы в коде (имена классов, файлов, API paths) MAY оставаться на
английском по конвенции Dart/Flutter.

**Обоснование**: единый язык документации для команды и стейкхолдеров проекта.

### VIII. Фирменная палитра

Визуальный стиль приложения MUST использовать утверждённую палитру osetrovich.ru. Цвета
MUST быть определены централизованно (например, `lib/core/theme/app_colors.dart`) и
применяться через `ThemeData` / design tokens, а не хардкодом в виджетах.

| Роль | HEX | Назначение |
|------|-----|------------|
| **Тёмный** | `#252A2F` | Основной текст, тёмные поверхности, иконки |
| **Primary** | `#213C57` | Основной брендовый цвет, app bar, ключевые элементы |
| **Accent** | `#FFB400` | Акценты, CTA-кнопки, индикаторы, выделение |
| **Background** | `#F4F5F5` | Фон экранов, светлые поверхности, карточки |

Отклонения от палитры MUST быть обоснованы в plan.md (Complexity Tracking). Произвольные
оттенки вне таблицы MUST NOT использоваться для брендовых элементов.

**Обоснование**: единая визуальная идентичность с сайтом магазина и узнаваемость бренда.

### IX. JWT-авторизация

Аутентификация и авторизация пользователей MUST выполняться через **JWT-токены**:

- Эндпоинты login/refresh/logout MUST быть описаны в OpenAPI (принцип VI).
- Access-токен MUST передаваться в заголовке `Authorization: Bearer <token>`.
- Токены MUST храниться в защищённом хранилище (`flutter_secure_storage` или эквивалент);
  MUST NOT храниться в plain `shared_preferences`.
- Истечение access-токена MUST обрабатываться: refresh-токен (если предусмотрен контрактом)
  или повторный вход пользователя.
- Моки авторизации MUST возвращать валидные JWT-структуры согласно OpenAPI-схеме.
- Сценарии авторизации (вход, выход, истечение сессии) MUST покрываться тестами (принцип III).

**Обоснование**: JWT — стандартный механизм для stateless REST API; единый подход к
безопасности на клиенте и бэкенде.

## Технический стек

| Компонент | Требование |
|-----------|------------|
| Клиент | Flutter (Android + iOS) |
| Язык UI | Русский |
| Навигация | Tab Bar (корневая) |
| Дизайн | Палитра: `#252A2F`, `#213C57`, `#FFB400`, `#F4F5F5` |
| Бэкенд | REST API (внешний) |

## Рабочий процесс и документация

1. Новая фича начинается с `/speckit-specify` → `spec.md` на русском языке.
2. План (`/speckit-plan`) MUST пройти **Constitution Check** до и после проектирования.
3. Задачи (`/speckit-tasks`) MUST включать тестовые задачи для основного функционала.
4. Реализация (`/speckit-implement`) MUST соблюдать принципы I–IX.
5. Изменения API: OpenAPI → моки → клиент → тесты контракта.
6. PR/review MUST проверять соответствие конституции; отклонения MUST быть задокументированы
   в разделе Complexity Tracking плана.

## Governance

- Конституция имеет приоритет над локальными договорённостями и шаблонами.
- Поправки вносятся через `/speckit-constitution` с обновлением зависимых шаблонов.
- **Версионирование** (semver):
  - **MAJOR** — удаление или переопределение принципа.
  - **MINOR** — новый принцип или существенное расширение раздела.
  - **PATCH** — уточнения формулировок без изменения смысла.
- Соответствие конституции проверяется в `/speckit-analyze`, `/speckit-converge` и при
  code review.
- Сложность сверх принципов MUST быть обоснована в plan.md (Complexity Tracking).
