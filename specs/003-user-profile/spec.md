# Спецификация фичи: Профиль пользователя

> **Язык**: этот документ и все артефакты Spec Kit MUST быть на **русском языке**
> (конституция проекта, принцип VII).

**Feature Branch**: `003-user-profile`

**Created**: 2026-07-14

**Status**: Implemented

**Input**: User description: "добавляем профиль: имя; email (с подтверждением кодом);
телефон (смена через подтверждение кодом); включение-выключение push-уведомлений;
ссылка на оферту https://osetrovich.ru/privacy-policy; кнопка связи с поддержкой
(как на главной); кнопки-иконки-ссылки на соцсети https://vk.com/osetrovich и
https://ok.ru/osetrovich; кнопка выхода."

## Изменения после реализации (2026-07-14)

- **PIN и биометрия сняты с scope**: локальный код входа, app lock и вход по биометрии
  не реализуются. Авторизация — только по телефону и СМС-коду (как в init-app-shell).
- **Сессия бессрочная на клиенте**: JWT-токены сохраняются в secure storage;
  `AuthSession.isExpired` всегда `false`; пользователь остаётся авторизованным до
  явного «Выйти» или переустановки приложения.
- **UI профиля**: кнопка «Выйти» — заливка `#FFB400`, текст `#252A2F`; блок
  «Связаться» — `ListTile` (как оферта), без иконки «открыть в новой вкладке»;
  иконки VK/OK — Font Awesome 6 Brands (`vk`, `odnoklassniki`), цвет `#252A2F`.
- **ContactBlock** (`ListTile`) — только в профиле (`LegalSupportSection`);
  на «Главной» — отдельный `HomeContactButton` (см. 002-notifications-home).
- **Иконки приложения** — из `assets/icons/` (Android adaptive, iOS, Web, macOS).
- **Мок профиля**: при восстановлении сессии `MockApiClient.ensureProfile()` подставляет
  профиль по телефону из access token.

## User Scenarios & Testing *(mandatory)*

### User Story 1 — Экран профиля и выход (Priority: P1)

Авторизованный покупатель открывает вкладку «Профиль» и видит свои данные: имя, email,
телефон. Может выйти из аккаунта одной кнопкой. Неавторизованный пользователь по-прежнему
видит заглушку с предложением войти (поведение из init-app-shell).

**Why this priority**: базовый экран профиля заменяет заглушку после авторизации; выход —
обязательный сценарий безопасности.

**Independent Test**: войти в приложение, открыть «Профиль», проверить отображение имени,
email и телефона; нажать «Выйти» и убедиться, что снова показывается заглушка авторизации.

**Acceptance Scenarios**:

1. **Given** пользователь авторизован, **When** он открывает вкладку «Профиль»,
   **Then** отображаются поля «Имя», «Email», «Телефон» с актуальными значениями из
   аккаунта; заглушка «Необходима авторизация» не показывается.
2. **Given** пользователь не авторизован, **When** он открывает «Профиль»,
   **Then** отображается заглушка с кнопкой входа (как в init-app-shell).
3. **Given** авторизованный пользователь на экране «Профиль», **When** он нажимает
   «Выйти», **Then** сессия завершается, пользователь остаётся на «Профиле» с заглушкой
   авторизации; защищённые разделы приложения требуют повторного входа.

---

### User Story 2 — Имя и смена телефона (Priority: P1)

Покупатель может изменить отображаемое имя и сменить номер телефона. Смена телефона
требует ввода нового номера и подтверждения шестизначным кодом из СМС (аналогично входу
по телефону).

**Why this priority**: телефон — основной идентификатор аккаунта; имя — базовая
персонализация.

**Independent Test**: изменить имя и сохранить; инициировать смену телефона, ввести код из
СМС и проверить, что новый номер отображается в профиле.

**Acceptance Scenarios**:

1. **Given** пользователь на «Профиле», **When** он редактирует и сохраняет имя,
   **Then** новое имя отображается на экране и сохраняется в аккаунте.
2. **Given** пользователь инициирует смену телефона, **When** он вводит новый номер
   в формате +7 и подтверждает, **Then** отправляется запрос на СМС-код и открывается
   экран ввода кода.
3. **Given** открыт экран подтверждения смены телефона, **When** пользователь вводит
   корректный шестизначный код, **Then** телефон в профиле обновляется на новый номер.
4. **Given** пользователь вводит неверный код при смене телефона, **When** подтверждение
   не проходит, **Then** отображается сообщение об ошибке на русском; старый номер
   остаётся в профиле.
5. **Given** экран ввода кода смены телефона, **When** пользователь нажимает «Назад»,
   **Then** он возвращается к предыдущему шагу без изменения номера.

---

### User Story 3 — Email с подтверждением кодом (Priority: P2)

Покупатель указывает или меняет email. После ввода адреса система отправляет код
подтверждения на email; email считается подтверждённым только после успешного ввода кода.

**Why this priority**: email нужен для коммуникаций и восстановления доступа, но не блокирует
базовый профиль и смену телефона.

**Independent Test**: ввести email, получить код (в тестовой среде — из мока), подтвердить
и проверить статус «подтверждён» в профиле.

**Acceptance Scenarios**:

1. **Given** у пользователя нет подтверждённого email, **When** он вводит новый адрес
   и запрашивает подтверждение, **Then** открывается шаг ввода кода, отправленного на
   указанный email.
2. **Given** пользователь вводит корректный код подтверждения email, **When** проверка
   успешна, **Then** email отображается в профиле как подтверждённый.
3. **Given** пользователь меняет email на другой адрес, **When** он проходит подтверждение
   кодом, **Then** в профиле отображается новый подтверждённый email.
4. **Given** введён некорректный формат email, **When** пользователь пытается продолжить,
   **Then** показывается понятная ошибка валидации на русском языке.
5. **Given** введён неверный код подтверждения email, **When** проверка не проходит,
   **Then** email не считается подтверждённым; пользователь может повторить ввод или
   запросить код повторно (с таймером, как при входе по СМС).

### User Story 4 — Push-уведомления (Priority: P3)

Покупатель включает или выключает push-уведомления из профиля. При первом включении
система запрашивает разрешение ОС, если оно ещё не выдано.

**Why this priority**: важно для удержания, но не блокирует остальной профиль.

**Independent Test**: переключить push в профиле, проверить сохранение состояния после
перезапуска приложения.

**Acceptance Scenarios**:

1. **Given** пользователь на «Профиле», **When** он видит переключатель push-уведомлений,
   **Then** состояние соответствует текущей настройке аккаунта/устройства.
2. **Given** push выключены, **When** пользователь включает переключатель,
   **Then** при необходимости запрашивается разрешение ОС; настройка сохраняется.
3. **Given** push включены, **When** пользователь выключает переключатель,
   **Then** приложение перестаёт отправлять push (настройка сохраняется).
4. **Given** пользователь отклонил разрешение ОС на push, **When** он пытается включить
   push, **Then** показывается понятное сообщение с предложением открыть настройки
   устройства (без технического жаргона).

### User Story 5 — Поддержка, оферта и соцсети (Priority: P3)

На экране «Профиль» доступны: звонок в поддержку (`ContactBlock`, `ListTile`), переход
к тексту оферты/политики на сайте, иконки соцсетей ВКонтакте и Одноклассники.

**Why this priority**: справочные и коммуникационные элементы; не блокируют основные
данные профиля.

**Independent Test**: нажать «Связаться» — инициируется звонок; нажать ссылку оферты —
открывается https://osetrovich.ru/privacy-policy; нажать иконки VK и OK — открываются
соответствующие страницы сообществ.

**Acceptance Scenarios**:

1. **Given** пользователь на «Профиле», **When** он нажимает «Связаться»,
   **Then** инициируется звонок на номер +7 812 564-55-48 (тот же номер, что на «Главной»);
   оформление — `ListTile` с иконкой телефона (без trailing «открыть ссылку»).
2. **Given** пользователь на «Профиле», **When** он нажимает ссылку на оферту,
   **Then** открывается страница https://osetrovich.ru/privacy-policy во внешнем
   браузере или встроенном просмотрщике.
3. **Given** пользователь на «Профиле», **When** он нажимает иконку ВКонтакте,
   **Then** открывается https://vk.com/osetrovich (иконка FA Brands `vk`, `#252A2F`).
4. **Given** пользователь на «Профиле», **When** он нажимает иконку Одноклассники,
   **Then** открывается https://ok.ru/osetrovich (иконка FA Brands `odnoklassniki`,
   `#252A2F`).
5. **Given** пользователь не авторизован, **When** он на «Профиле» с заглушкой,
   **Then** блоки поддержки, оферты и соцсетей MAY быть доступны без входа
   (рекомендуется: да, для юридической информации и связи).

### Edge Cases

- Что происходит при потере сети при сохранении имени или подтверждении кодов? —
  Показывается сообщение об ошибке на русском; данные не меняются частично без подтверждения
  сервера.
- Что если email уже занят другим аккаунтом? — Сообщение «Этот email уже используется»;
  текущий email профиля не меняется.
- Что если новый телефон уже зарегистрирован? — Сообщение об ошибке; смена не выполняется.
- Истечение таймера повторной отправки кода (email/телефон)? — Кнопка «Отправить код
  повторно» активна после 60 секунд (как при входе).
- Перезапуск приложения с сохранённой сессией? — Пользователь остаётся авторизованным;
  профиль загружается из API/мока без повторного входа по СМС.
- Выход из аккаунта? — JWT-сессия очищается; при следующем входе — телефон + СМС.
- Push: пользователь включил в профиле, но запретил в ОС? — Переключатель отражает
  фактическое состояние доставки; подсказка открыть настройки устройства.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: Вкладка «Профиль» MUST отображать данные авторизованного пользователя:
  имя, email (со статусом подтверждения), телефон.
- **FR-002**: Для неавторизованного пользователя MUST сохраняться заглушка входа из
  init-app-shell.
- **FR-003**: Пользователь MUST иметь возможность изменить и сохранить имя в профиле.
- **FR-004**: Смена телефона MUST выполняться через ввод нового номера +7 и подтверждение
  шестизначным СМС-кодом.
- **FR-005**: Указание или смена email MUST выполняться с подтверждением кодом,
  отправленным на указанный адрес.
- **FR-006**: Email MUST считаться подтверждённым только после успешной проверки кода.
- **FR-007**: ~~Локальный PIN и биометрия~~ — **снято с scope** (не реализуется).
- **FR-008**: JWT-сессия MUST сохраняться в secure storage и восстанавливаться при
  перезапуске приложения без истечения срока на клиенте.
- **FR-009**: Пользователь MUST иметь переключатель включения/выключения push-уведомлений
  в профиле.
- **FR-010**: MUST быть ссылка на оферту/политику: https://osetrovich.ru/privacy-policy.
- **FR-011**: MUST быть кнопка «Связаться» с инициацией звонка на +7 812 564-55-48
  (оформление `ListTile`, как оферта).
- **FR-012**: MUST быть иконки-ссылки на https://vk.com/osetrovich и
  https://ok.ru/osetrovich (Font Awesome 6 Brands, цвет `#252A2F`).
- **FR-013**: MUST быть кнопка «Выйти» (заливка `#FFB400`, текст `#252A2F`),
  завершающая JWT-сессию и возвращающая профиль в состояние неавторизованного пользователя.
- **FR-014**: Все тексты, ошибки и подсказки на экранах профиля MUST быть на русском языке.
- **FR-015**: Экраны ввода кодов (телефон, email) MUST поддерживать «Назад», повторную
  отправку кода с таймером 60 секунд и сообщения об ошибках, согласованные с экранами
  авторизации.

### Key Entities

- **UserProfile**: имя, email, флаг `emailVerified`, телефон; связь с JWT-сессией.
- **VerificationChallenge**: тип (sms | email), целевой адрес (телефон/email), срок действия
  кода, возможность повторной отправки.
- **AuthSession**: access/refresh token, телефон; бессрочная на клиенте (`isExpired: false`).
- **NotificationPreferences**: флаг push-enabled (синхронизация с сервером и разрешением ОС).

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: Авторизованный пользователь видит имя, email и телефон на «Профиле» не
  более чем за 2 секунды после открытия вкладки.
- **SC-002**: Смена телефона с корректным кодом завершается за один проход (≤ 3 минут
  включая ожидание СМС) у 90% пользователей.
- **SC-003**: Подтверждение email с корректным кодом завершается за один проход у 90%
  пользователей.
- **SC-004**: Переключение push сохраняет состояние после перезапуска приложения в 100%
  случаев успешного сохранения на сервере.
- **SC-005**: Звонок в поддержку, оферта и соцсети открываются с первого нажатия в 100%
  случаев при наличии сети/приложений на устройстве.
- **SC-006**: Выход из аккаунта завершает сессию за ≤ 2 секунды; повторное открытие
  «Профиля» показывает заглушку авторизации.
- **SC-007**: После перезапуска приложения авторизованный пользователь остаётся в сессии
  без повторного ввода СМС-кода.

## Assumptions

- Профиль расширяет заглушку из `001-init-app-shell`; авторизация по телефону и СМС уже
  реализована.
- Код подтверждения email — шестизначный, аналогичный СМС-коду входа; срок жизни и лимиты
  повторной отправки — как у auth-flow (60 с таймер).
- JWT-токены хранятся в `flutter_secure_storage`; сессия на клиенте не истекает
  (`AuthSession.neverExpiresAt`, `isExpired` всегда `false`) до явного «Выйти».
- При «Выйти» JWT-сессия сбрасывается; для повторного доступа нужен вход по телефону и СМС.
- Push-уведомления: переключатель в профиле синхронизируется с бэкендом; фактическая
  доставка зависит от разрешения ОС (Android/iOS).
- Ссылка «оферта» ведёт на URL https://osetrovich.ru/privacy-policy (единая юридическая
  страница на сайте магазина).
- На «Главной» — `HomeContactButton` (серая баннер-кнопка); в профиле — `ContactBlock`
  (`ListTile`). Оба инициируют звонок на один номер, но UI разделены.
- Соцсети открываются во внешнем браузере или соответствующем приложении (VK, OK).
- API профиля и верификации будет описан в OpenAPI на этапе планирования (принцип VI
  конституции); до бэкенда — моки.
- Редактирование заказов, адресов доставки и истории покупок — вне scope данной фичи.
