# Спецификация фичи: Автоматическое обновление токена при 401

> **Язык**: этот документ и все артефакты Spec Kit MUST быть на **русском языке**
> (конституция проекта, принцип VII).

**Feature Branch**: `010-token-refresh-401`

**Created**: 2026-07-20

**Status**: Draft

**Input**: User description: "Если бекенд отвечает 401, то пытаться обновить токен с помощью refreshToken. Сейчас в приложении выводится ошибка DioException[bad response]. Если обновить токен не удалось, то разавторизовывать пользователя в приложении."

## User Scenarios & Testing *(mandatory)*

### User Story 1 — Прозрачное обновление сессии при истечении access-токена (Priority: P1)

Авторизованный покупатель продолжает пользоваться приложением (каталог, корзина, профиль,
история заказов). Access-токен истёк, и сервер отвечает «не авторизован» (401). Приложение
автоматически обновляет сессию с помощью сохранённого refresh-токена и повторяет исходный
запрос. Пользователь не видит технических сообщений об ошибке и не замечает сбоя.

**Why this priority**: это основной сценарий, ради которого вводится фича — непрерывная работа
без принудительного выхода при каждом истечении access-токена.

**Independent Test**: авторизоваться, симулировать истечение access-токена (или получить 401
от защищённого эндпоинта), выполнить действие (например, открыть профиль или корзину) и
убедиться, что данные загружаются без сообщения об ошибке и без перехода на экран входа.

**Acceptance Scenarios**:

1. **Given** пользователь авторизован и access-токен недействителен, **When** приложение
   выполняет защищённый запрос к серверу и получает ответ 401, **Then** приложение
   автоматически запрашивает новую пару токенов с помощью refresh-токена, сохраняет их
   и повторяет исходный запрос с новым access-токеном.
2. **Given** обновление токена прошло успешно, **When** исходный запрос повторён,
   **Then** пользователь видит ожидаемый результат (данные, экран, подтверждение действия)
   без технических сообщений об ошибке.
3. **Given** пользователь авторизован, **When** несколько запросов одновременно получают 401,
   **Then** обновление токена выполняется один раз, после чего все ожидающие запросы
   повторяются с новым access-токеном.

### User Story 2 — Выход при невозможности обновить сессию (Priority: P1)

Авторизованный покупатель пытается выполнить действие, требующее авторизации, но refresh-токен
тоже недействителен или отклонён сервером. Приложение завершает сессию: очищает сохранённые
токены и переводит пользователя в состояние «не авторизован» с понятным предложением войти
снова.

**Why this priority**: безопасное завершение сессии при невалидных учётных данных — обязательное
требование; иначе пользователь остаётся в «полусломанном» состоянии с бессмысленными ошибками.

**Independent Test**: авторизоваться, симулировать отказ сервера на обновление токена (401 или
эквивалент «сессия недействительна»), выполнить защищённое действие и убедиться, что
пользователь разлогинен и видит заглушку/экран входа, а не техническую ошибку.

**Acceptance Scenarios**:

1. **Given** пользователь авторизован, **When** защищённый запрос получает 401 и попытка
   обновить токен завершается неуспехом (сервер отклоняет refresh-токен), **Then** сессия
   пользователя завершается: токены удаляются из хранилища, состояние приложения
   соответствует неавторизованному пользователю.
2. **Given** сессия завершена из-за неуспешного обновления токена, **When** пользователь
   находится на защищённом разделе (профиль, корзина с персональными данными и т.п.),
   **Then** отображается предложение войти (как для неавторизованного пользователя), а не
   сообщение вида «DioException» или «bad response».
3. **Given** сессия завершена из-за неуспешного обновления, **When** пользователь пытается
   выполнить действие, требующее авторизации, **Then** приложение направляет на сценарий
   входа по телефону.

### User Story 3 — Понятные сообщения вместо технических ошибок (Priority: P2)

Покупатель сталкивается с проблемой авторизации или сети. Вместо внутренних технических
сообщений (например, «DioException[bad response]») приложение показывает короткое
понятное сообщение на русском языке или стандартное пустое/ошибочное состояние экрана
в соответствии с контекстом.

**Why this priority**: улучшает доверие и снижает обращения в поддержку; не блокирует
основную логику refresh/logout, но закрывает заявленную боль пользователя.

**Independent Test**: вызвать сценарий с ошибкой авторизации или сети на любом экране с
загрузкой данных и убедиться, что в UI нет строк с «DioException», «bad response» и
аналогичными техническими деталями.

**Acceptance Scenarios**:

1. **Given** произошла ошибка авторизации после неуспешного обновления токена, **When**
   пользователь видит результат на экране, **Then** отображается человекочитаемое сообщение
   или состояние «необходимо войти», без технических деталей HTTP-клиента.
2. **Given** произошла сетевая ошибка при обновлении токена (сервер недоступен, таймаут),
   **When** пользователь видит уведомление об ошибке, **Then** текст на русском языке
   описывает суть проблемы («Нет соединения», «Не удалось загрузить данные» и т.п.), без
   имён классов исключений и кодов библиотек.

---

### Edge Cases

- Запрос к эндпоинту обновления токена сам возвращает 401 — сессия завершается, повторного
  обновления не предпринимается (избежание бесконечного цикла).
- 401 на публичных или гостевых эндпоинтах (каталог без авторизации) — обновление токена
  не запускается, если пользователь не был авторизован.
- 401 на эндпоинтах входа (запрос/подтверждение СМС) — обновление токена не применяется;
  показывается стандартная ошибка сценария входа.
- Пользователь нажимает «Выйти» во время фонового обновления токена — выход имеет приоритет,
  отложенные повторные запросы не восстанавливают сессию.
- Refresh-токен отсутствует в хранилище при 401 — сессия считается завершённой, пользователь
  разлогинивается без попытки обновления.
- Ошибки, отличные от 401 (403, 404, 500) — не запускают обновление токена; обрабатываются
  как обычные ошибки соответствующего экрана.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: Приложение MUST при получении ответа 401 на защищённый запрос (при наличии
  сохранённого refresh-токена) автоматически запрашивать новую пару токенов у сервера.
- **FR-002**: После успешного обновления токенов приложение MUST сохранить новые access- и
  refresh-токены в защищённом хранилище и обновить состояние сессии пользователя.
- **FR-003**: После успешного обновления приложение MUST автоматически повторить исходный
  запрос(ы), прерванные из-за 401, с новым access-токеном.
- **FR-004**: При неуспешном обновлении токена (отказ сервера, недействительный refresh-токен)
  приложение MUST завершить сессию: очистить токены и перевести пользователя в состояние
  неавторизованного.
- **FR-005**: Параллельные запросы, получившие 401, MUST координироваться так, чтобы
  обновление токена выполнялось не более одного раза на «волну» истечения сессии.
- **FR-006**: Запрос обновления токена MUST NOT инициировать повторное обновление при
  получении 401 на самом эндпоинте обновления (защита от цикла).
- **FR-007**: Пользовательский интерфейс MUST NOT отображать технические сообщения HTTP-клиента
  (включая «DioException», «bad response») при ошибках, связанных с авторизацией; вместо
  этого — понятные русскоязычные формулировки или штатные заглушки экранов.
- **FR-008**: Сценарии автоматического обновления и принудительного выхода MUST применяться
  ко всем защищённым операциям приложения (профиль, корзина, заказы, оформление и др.).
- **FR-009**: Поведение при обновлении токена MUST соответствовать контракту API обновления
  сессии (запрос с refresh-токеном, ответ с новой парой токенов).

### Key Entities

- **Сессия пользователя**: связка access-токена, refresh-токена и данных аккаунта; определяет,
  авторизован ли покупатель в приложении.
- **Пара токенов**: access-токен (краткоживущий, для запросов) и refresh-токен (для
  продления сессии); выдаётся при входе и при успешном обновлении.
- **Защищённый запрос**: обращение к серверу, требующее действующей авторизации; при 401
  запускает логику обновления сессии.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: В 95% сценариев с истёкшим access-токеном авторизованный пользователь
  завершает целевое действие (просмотр профиля, корзины, оформление) без ручного повторного
  входа и без видимой ошибки на экране.
- **SC-002**: При недействительном refresh-токене пользователь оказывается в состоянии
  «не авторизован» не более чем за 3 секунды после первого неуспешного защищённого запроса.
- **SC-003**: 0% отображаемых пользователю сообщений об ошибках авторизации содержат
  технические строки HTTP-клиента (проверка по чек-листу основных экранов: профиль, корзина,
  главная с заказами).
- **SC-004**: Количество обращений в поддержку с формулировками «DioException» / «bad response»
  при работе с авторизованными разделами снижается до нуля после релиза фичи (качественная
  метрика на период первого месяца после выката).

## Assumptions

- Контракт API уже описывает эндпоинт обновления токена и формат тела запроса/ответа;
  клиент использует существующий механизм хранения токенов в защищённом хранилище.
- Истечение access-токена на стороне сервера проявляется ответом 401 на защищённые эндпоинты;
  другие коды (например, 403) не трактуются как сигнал к обновлению токена.
- При успешном обновлении сервер возвращает новую пару access- и refresh-токенов; клиент
  заменяет оба значения.
- Сетевые сбои при обновлении токена (таймаут, отсутствие сети) не приравниваются к
  «недействительной сессии»: пользователь видит сообщение о проблеме с соединением, сессия
  сохраняется до явного отказа сервера на refresh или выхода пользователя.
- Поведение для неавторизованного пользователя при 401 на гостевых сценариях не меняется:
  предлагается войти без попытки refresh.
- Автоматизированные тесты (unit, widget, integration) покроют основные сценарии обновления
  и выхода в соответствии с конституцией проекта (принципы III и IX).
