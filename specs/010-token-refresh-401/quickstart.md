# Quickstart: Автоматическое обновление токена при 401

**Фича**: [spec.md](./spec.md) | **План**: [plan.md](./plan.md) | **Модель**: [data-model.md](./data-model.md)

Руководство по проверке фичи после реализации.

## Предварительные требования

- Flutter SDK (stable), Dart 3.x
- Репозиторий с выполненными задачами из `tasks.md`
- Для ручной проверки на бэкенде: `useMockApi = false` в `lib/core/network/providers.dart`
- Для unit-тестов интерцептора: не требуется боевой API

## Установка и запуск

```bash
flutter pub get
flutter run
```

## Автоматические тесты

```bash
flutter test test/core/network/token_refresh_interceptor_test.dart
flutter test test/core/network/api_error_mapper_test.dart
flutter test test/features/auth/auth_session_refresh_test.dart
flutter test test/features/profile/profile_screen_test.dart
flutter analyze
```

**Ожидание**:
- Интерцептор: 401 → refresh → retry успешен; параллельные 401 — один refresh
- Refresh 401 → вызван `onSessionExpired`, сессия очищена
- Сетевой сбой при refresh — сессия сохранена, `NETWORK_ERROR` на русском
- Profile error state не содержит `DioException`

## Сценарии ручной проверки

### С1. Прозрачный refresh (US1, FR-001–FR-003)

**Предусловие**: пользователь авторизован, `useMockApi = false`.

1. Дождаться истечения access-токена **или** временно подменить access в secure storage
   на недействительное значение (через dev-инструмент / отладку).
2. Открыть «Профиль» или «Корзину» (защищённый запрос).
3. Данные загружаются без ошибки; пользователь остаётся авторизованным.
4. В логах (debug) виден один вызов `/auth/refresh` и повтор исходного запроса.

### С2. Logout при мёртвом refresh (US2, FR-004)

1. Авторизоваться.
2. Подменить **оба** токена на недействительные **или** дождаться отзыва refresh на сервере.
3. Открыть «Профиль».
4. Пользователь разлогинен; отображается заглушка «Необходима авторизация» / кнопка входа.
5. Текст ошибки **не** содержит `DioException` или `bad response`.

### С3. Параллельные запросы (FR-005)

1. Авторизоваться с истёкшим access.
2. Открыть «Главную» (несколько защищённых запросов: заказ, уведомления).
3. Экран загружается корректно; в network trace — **один** `POST /auth/refresh`.

### С4. Сетевая ошибка при refresh (Assumptions)

1. Авторизоваться; включить режим полёта **после** получения 401 (сложно вручную) **или**
   использовать unit-тест как эталон.
2. При сетевом сбое во время refresh пользователь **не** разлогинен.
3. Показано сообщение о проблеме с соединением на русском.

### С5. Явный выход vs фоновый refresh (Edge case)

1. Авторизоваться; инициировать действие с 401 (refresh в полёте).
2. Сразу нажать «Выйти» в профиле.
3. Сессия завершена; повторные запросы не восстанавливают авторизацию.

### С6. Вход по СМС не затронут (Edge case)

1. Выйти из аккаунта.
2. Начать вход: неверный код СМС.
3. Отображается штатная ошибка входа; **нет** вызова `/auth/refresh`.

### С7. Мок-режим (регрессия)

1. `useMockApi = true`.
2. Приложение работает как раньше; тесты auth/profile зелёные.
