## Clarifications

### Session 2026-07-20

- Q: Каков срок доступности оценки заказа? → A: 7 суток с даты доставки (`delivery_at`); после
  истечения оценить или пропустить нельзя; сервер возвращает HTTP 400 с кодом
  `rating_period_expired`.

### Session 2026-07-20 (UX polish)

- После успешной отправки оценки — SnackBar «Спасибо за оценку!».
- Кнопка «Оценить заказ» в уведомлении «Заказ доставлен» скрывается реактивно, когда
  `ratingState != pending` (в т.ч. сразу после успешной оценки).
- Поле комментария в форме оценки оформлено как в корзине (`OutlineInputBorder`, подпись
  «Комментарий к заказу», без hint «Необязательно»); первая буква — заглавная
  (`TextCapitalization.sentences`).
- Pull-to-refresh на «Главной» обновляет список in-app уведомлений (не только счётчик).
- Категория «Все» (`id: 0`) принудительно добавляется первой в списке категорий каталога
  на клиенте, даже если API её не возвращает.

## User Scenarios & Testing *(mandatory)*

### User Story 1 — Актуальный текущий заказ с сервера (Priority: P1)

Авторизованный покупатель на вкладке «Главная» видит блок «История заказов» с данными
текущего заказа, полученными с сервера магазина: статус, состав позиций, суммы, адрес
доставки и состояние оценки. Данные соответствуют реальному заказу покупателя в системе
магазина, а не демонстрационным или локальным заглушкам.

**Why this priority**: без достоверных данных о заказе главный экран вводит пользователя в
заблуждение; это базовый сценарий после перехода на боевой API.

**Independent Test**: авторизоваться под пользователем с активным или недавно завершённым
заказом на сервере, открыть «Главную» и убедиться, что статус, состав и сумма совпадают с
данными в личном кабинете или ERP магазина.

**Acceptance Scenarios**:

1. **Given** пользователь авторизован и на сервере есть актуальный заказ, **When** он
   открывает «Главную», **Then** в блоке «История заказов» отображаются статус, позиции,
   итоговая сумма и адрес доставки из ответа сервера.
2. **Given** пользователь авторизован, но на сервере нет актуального заказа (`order: null`),
   **When** он открывает «Главную», **Then** блок «История заказов» не отображается.
3. **Given** пользователь не авторизован, **When** он открывает «Главную»,
   **Then** блок «История заказов» не отображается и запрос текущего заказа не выполняется.
4. **Given** пользователь оформил новый заказ, **When** он возвращается на «Главную» или
   обновляет экран, **Then** блок заказа показывает только что созданный заказ с сервера.
5. **Given** статус заказа изменился на сервере (например, с «Доставка» на «Выполнено»),
   **When** пользователь обновляет «Главную», **Then** отображается актуальный статус.
6. **Given** запрос текущего заказа завершился ошибкой сети или сервера, **When** пользователь
   смотрит на «Главную», **Then** он видит понятное сообщение об ошибке на русском языке и
   может повторить загрузку.

### User Story 2 — Оценка выполненного заказа с сохранением на сервере (Priority: P1)

Авторизованный покупатель с выполненным заказом, по которому сервер ожидает оценку
(`ratingState: pending`), видит приглашение оценить заказ. После выбора оценки от 1 до 5
звёзд и необязательного комментария оценка сохраняется на сервере; приглашение исчезает,
появляется возможность повторить заказ.

**Why this priority**: оценка — ключевая обратная связь для магазина; без сохранения на
сервере данные теряются при смене устройства или переустановке приложения.

**Independent Test**: открыть «Главную» с выполненным неоценённым заказом на сервере,
отправить оценку 4 звезды с комментарием, обновить экран и убедиться, что приглашение к
оценке не показывается повторно.

**Acceptance Scenarios**:

1. **Given** текущий заказ в статусе «Выполнено» и `ratingState` равен `pending`,
   **When** пользователь открывает «Главную», **Then** отображается приглашение к оценке с
   кнопками «Оценить» и «Пропустить».
2. **Given** пользователь нажал «Оценить», **When** он выбирает от 1 до 5 звёзд и
   подтверждает отправку, **Then** оценка сохраняется на сервере; приглашение скрывается;
   отображается кнопка «Повторить заказ».
3. **Given** пользователь отправляет оценку с комментарием, **When** комментарий не превышает
   допустимую длину, **Then** оценка и комментарий сохраняются на сервере.
4. **Given** пользователь отправляет оценку без комментария, **When** он подтверждает
   отправку, **Then** оценка сохраняется успешно.
5. **Given** сервер вернул ошибку «оценка уже отправлена или пропущена», **When** пользователь
   пытается оценить заказ повторно, **Then** он видит понятное сообщение на русском языке;
   интерфейс обновляется в соответствии с актуальным состоянием заказа с сервера.
6. **Given** заказ ещё не выполнен, **When** пользователь пытается оценить его,
   **Then** оценка недоступна; пользователь видит сообщение «Оценка недоступна» или
   приглашение к оценке не отображается.
7. **Given** с даты доставки прошло более 7 суток и оценка не была отправлена или пропущена,
   **When** пользователь пытается оценить или пропустить заказ, **Then** сервер отклоняет
   действие; пользователь видит понятное сообщение о том, что срок оценки истёк.
8. **Given** с даты доставки прошло более 7 суток и оценка не была отправлена или пропущена,
   **When** пользователь открывает «Главную», **Then** выполненный неоценённый заказ не
   отображается как текущий (сервер возвращает `order: null` или не показывает prompt).

### User Story 3 — Пропуск оценки с фиксацией на сервере (Priority: P2)

Покупатель может отказаться от оценки выполненного заказа. Решение о пропуске сохраняется
на сервере; приглашение к оценке больше не показывается; доступна кнопка «Повторить заказ».

**Why this priority**: пользователь должен иметь возможность закрыть сценарий оценки без
отправки звёзд; состояние должно быть единым на всех устройствах.

**Independent Test**: для выполненного заказа с `ratingState: pending` нажать «Пропустить»,
обновить «Главную» и убедиться, что приглашение не возвращается.

**Acceptance Scenarios**:

1. **Given** отображается приглашение к оценке, **When** пользователь нажимает «Пропустить»,
   **Then** пропуск фиксируется на сервере; приглашение скрывается; кнопка «Повторить заказ»
   становится доступной.
2. **Given** пользователь уже пропустил оценку, **When** он снова открывает «Главную»,
   **Then** приглашение к оценке не отображается.
3. **Given** сервер вернул ошибку при пропуске (заказ не найден, оценка уже зафиксирована
   или истёк срок оценки), **When** пользователь нажимает «Пропустить», **Then** отображается
   понятное сообщение об ошибке; состояние экрана синхронизируется с сервером.

### User Story 4 — Оценка заказа из уведомления (Priority: P2)

Покупатель, получивший уведомление о необходимости оценить заказ, может перейти к оценке из
экрана уведомления. Оценка сохраняется на сервере по тем же правилам, что и с «Главной».

**Why this priority**: уведомления — альтернативная точка входа в сценарий оценки; поведение
должно быть согласованным.

**Independent Test**: открыть уведомление с действием «Оценить заказ» при `ratingState:
pending`, отправить оценку и проверить обновление блока заказа на «Главной».

**Acceptance Scenarios**:

1. **Given** у пользователя есть выполненный заказ с `ratingState: pending`, **When** он
   открывает соответствующее уведомление и нажимает «Оценить заказ», **Then** открывается
   интерфейс оценки; после отправки оценка сохраняется на сервере, отображается «Спасибо за
   оценку!», кнопка «Оценить заказ» на экране уведомления скрывается.
2. **Given** оценка уже отправлена, пропущена или истёк срок оценки (7 суток с даты доставки),
   **When** пользователь открывает уведомление «Заказ доставлен», **Then** кнопка «Оценить
   заказ» не отображается; при попытке оценить через устаревший UI — сообщение «Оценка
   недоступна» или о сроке истечения оценки.

---

### Edge Cases

- Сессия истекла во время загрузки заказа или отправки оценки — пользователь перенаправляется
  на повторный вход; после входа данные подгружаются заново.
- Комментарий к оценке длиннее допустимого лимита — отправка блокируется или пользователь
  получает понятное сообщение до отправки на сервер.
- Заказ с `ratingState: not_applicable` (активная доставка) — приглашение к оценке не
  показывается.
- Заказ с `ratingState: submitted` — отображается выполненный заказ с кнопкой «Повторить
  заказ», без приглашения к оценке.
- Позиции заказа содержат товары без изображений — состав заказа отображается корректно.
- Одновременное нажатие «Оценить» / «Пропустить» — повторные запросы не создают дублирующих
  действий; кнопки блокируются на время отправки.
- Истёк срок оценки (более 7 суток с даты доставки) — оценка и пропуск недоступны; сервер
  возвращает ошибку `rating_period_expired`; приглашение к оценке не показывается; для
  неоценённого выполненного заказа `GET /orders/current` может вернуть `order: null`.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: Для авторизованных пользователей приложение MUST запрашивать текущий заказ с
  сервера магазина и отображать его в блоке «История заказов» на «Главной».
- **FR-002**: При отсутствии актуального заказа на сервере блок «История заказов» MUST быть
  скрыт.
- **FR-003**: Для неавторизованных пользователей запрос текущего заказа MUST NOT выполняться.
- **FR-004**: Отображаемые статус, состав, суммы, адрес доставки и состояние оценки MUST
  соответствовать данным, возвращаемым сервером.
- **FR-005**: В рабочей конфигурации приложения текущий заказ и оценка MUST получаться и
  сохраняться через Mobile API, а не через локальные демонстрационные данные.
- **FR-006**: Пользователь MUST иметь возможность отправить оценку от 1 до 5 звёзд с
  необязательным комментарием для выполненного заказа в состоянии `pending` **в течение 7
  суток с даты доставки**.
- **FR-007**: Пользователь MUST иметь возможность пропустить оценку выполненного заказа в
  состоянии `pending` **в течение 7 суток с даты доставки**.
- **FR-008**: После успешной оценки или пропуска приглашение к оценке MUST скрываться;
  MUST отображаться кнопка «Повторить заказ» (согласно существующему сценарию фичи 007).
- **FR-009**: При ошибках сервера (недоступность оценки, заказ не найден, оценка уже
  зафиксирована, **истёк срок оценки**) пользователь MUST видеть понятные сообщения на
  русском языке.
- **FR-010**: При ошибке загрузки текущего заказа пользователь MUST видеть сообщение об
  ошибке и возможность повторить попытку.
- **FR-011**: Оценка заказа MUST быть доступна как с «Главной», так и из экрана уведомления
  (при наличии соответствующего действия).
- **FR-012**: После оформления нового заказа, смены пользователя или обновления экрана блок
  заказа MUST отражать актуальные серверные данные.
- **FR-013**: После истечения 7 суток с даты доставки приложение MUST NOT предлагать оценить
  или пропустить заказ; попытка отправки MUST обрабатываться с сообщением об истечении срока.
- **FR-014**: После успешной отправки оценки пользователь MUST видеть сообщение благодарности
  («Спасибо за оценку!»).
- **FR-015**: На экране уведомления «Заказ доставлен» кнопка «Оценить заказ» MUST
  отображаться только при `ratingState: pending` и MUST скрываться сразу после успешной
  оценки.

### Key Entities

- **Текущий заказ**: актуальный заказ покупателя — идентификатор, номер, статус, позиции,
  суммы, адрес доставки, квартира (если есть), комментарий, дата создания, **дата доставки**
  (`delivery_at`), состояние оценки (`not_applicable`, `pending`, `submitted`, `skipped`),
  количество звёзд (если оценён).
- **Оценка заказа**: число звёзд от 1 до 5 и необязательный комментарий, привязанные к
  конкретному заказу; состояние хранится на сервере и определяет отображение UI. Доступна
  **7 суток с даты доставки**; после истечения — ошибка `rating_period_expired`.
- **Ответ «текущий заказ»**: обёртка с полем `order`, которое может быть `null`, если
  актуального заказа нет.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 100% авторизованных пользователей с заказом на сервере видят на «Главной»
  данные, совпадающие с сервером (проверка на тестовых аккаунтах с активным и завершённым
  заказом).
- **SC-002**: 95% попыток отправить оценку или пропустить её завершаются успешно при стабильном
  соединении; после успеха приглашение не появляется повторно при обновлении экрана.
- **SC-003**: Пользователь может отправить оценку или пропустить её не более чем за 30 секунд
  от открытия приглашения (без учёта времени ввода длинного комментария).
- **SC-004**: При ошибке загрузки заказа пользователь может восстановить отображение данных
  повторной попыткой без перезапуска приложения.
- **SC-005**: Сценарии оценки с «Главной» и из уведомления дают одинаковый итоговый статус
  заказа на сервере.
- **SC-006**: 100% попыток оценить или пропустить заказ после истечения 7 суток с даты
  доставки корректно отклоняются с понятным сообщением пользователю.

## Assumptions

- Обновлённая OpenAPI-спецификация Mobile API (v0.12.1) является источником истины для
  эндпоинтов текущего заказа и оценки.
- Оценка и пропуск доступны **7 суток с даты доставки** (`delivery_at`); сервер — источник
  истины для проверки срока; клиент не вычисляет TTL самостоятельно.
- При истечении срока сервер возвращает HTTP 400 с кодом `rating_period_expired`; выполненный
  неоценённый заказ может перестать возвращаться в `GET /orders/current`.
- Пользовательский интерфейс блока заказа, формы оценки и кнопки «Повторить заказ» уже
  реализован в рамках фичи 007; в scope этой фичи — подключение к боевому API и отказ от
  демонстрационных данных для текущего заказа.
- Оценка доступна только для заказов в статусе «Выполнено»; сервер определяет `ratingState`.
- Максимальная длина комментария к оценке — 500 символов (согласно контракту API).
- Демонстрационные заказы по тестовым номерам телефонов в мок-клиенте **удалены**; заказы в
  моке создаются только через `createOrder`.
- Зависимости: JWT-авторизация (фича 009/010), оформление заказа через API (фича 012).
