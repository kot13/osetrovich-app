# Quickstart: Текущий заказ и оценка через Mobile API

**Фича**: [spec.md](./spec.md) | **План**: [plan.md](./plan.md) | **Модель**: [data-model.md](./data-model.md)

Руководство по проверке фичи после реализации.

## Предварительные требования

- Flutter SDK (stable), Dart 3.x
- Репозиторий с выполненными задачами из `tasks.md`
- OpenAPI v0.12.1 в `openapi/openapi.yaml` (TTL 7 суток, `rating_period_expired`)
- Боевой API: `useMockApi = false`, base URL `https://trout.osetrovich.ru/v1`
- Авторизованная сессия (СМС-вход)
- UI блока заказа и оценки реализован (`007-home-screen-content`)

## Установка и запуск

```bash
flutter pub get
flutter run
```

Для локальных тестов с моком:

```bash
# временно в lib/core/network/providers.dart: useMockApi = true
flutter run
```

## Сценарии ручной проверки (боевой API)

### С1. Текущий заказ на «Главной» (US1)

1. Авторизоваться под пользователем с активным заказом на сервере.
2. Открыть «Главную».
3. Убедиться: блок «История заказов» показывает статус, состав и сумму, совпадающие с ERP.
4. Потянуть вниз (pull-to-refresh) — данные обновляются; список in-app уведомлений и счётчик
   непрочитанных синхронизируются с сервером.

---

### С2. Нет текущего заказа (US1, FR-002)

1. Авторизоваться под пользователем без активного заказа.
2. Открыть «Главную».
3. Блок «История заказов» не отображается.

---

### С3. Заказ после оформления (US1, FR-012)

1. Оформить новый заказ из «Корзины».
2. Перейти на «Главную».
3. Отображается только что созданный заказ (без ручного refresh).

---

### С4. Оценка заказа (US2)

1. Дождаться статуса «Выполнено» и `ratingState: pending` на сервере.
2. На «Главной» — приглашение к оценке, кнопки «Оценить» и «Пропустить».
3. Нажать «Оценить», выбрать 5 звёзд, добавить комментарий, отправить.
4. Появилось сообщение «Спасибо за оценку!»; приглашение исчезло; появилась кнопка «Повторить заказ».
5. Обновить экран — приглашение не возвращается.

---

### С5. Пропуск оценки (US3)

1. Для другого выполненного заказа с `pending` нажать «Пропустить».
2. Приглашение исчезло; «Повторить заказ» доступна.
3. После refresh приглашение не показывается.

---

### С6. Оценка из уведомления (US4)

1. Открыть уведомление «Заказ доставлен» с действием «Оценить заказ» (при `ratingState: pending`).
2. Отправить оценку — появляется «Спасибо за оценку!»; кнопка «Оценить заказ» исчезает.
3. На «Главной» состояние заказа согласовано (нет повторного prompt).

---

### С7. Ошибка загрузки заказа (US1, FR-010)

1. При недоступности API (или airplane mode) открыть «Главную».
2. Отображается «Не удалось загрузить данные» и кнопка «Повторить».
3. Восстановить сеть, нажать «Повторить» — блок заказа загружается.

---

### С8. Повторная оценка (edge case)

1. После успешной оценки попытаться оценить снова (если UI позволяет через уведомление).
2. Сообщение «Оценка недоступна» или «Оценка уже отправлена»; UI синхронизирован с сервером.

---

### С9. Истёк срок оценки (US2, FR-013, SC-006)

1. Взять выполненный заказ с `ratingState: pending`, доставленный более 7 суток назад.
2. На «Главной» блок заказа не отображается (или нет приглашения к оценке).
3. При попытке оценить из уведомления — сообщение «Срок оценки истёк».
4. `POST /rating` и `/rating/skip` возвращают 400 `rating_period_expired`.

## Сценарии с моком (для разработки)

### М1. Без demo phones

1. `useMockApi = true`.
2. Войти под любым номером — блок заказа **не** появляется (нет автосида).
3. Оформить заказ — на «Главной» появился заказ из `createOrder`.

### М2. Рейтинг в тестах

1. В unit-тесте: `createOrder` → `completeOrderForRating` → `submitOrderRating`.
2. `submitOrderRating` → `ratingState: submitted`.
3. Повторный submit → `ApiException` code `rating_already_set`.

### М3. TTL в моке

1. `completeOrderForRating` → `expireOrderRatingPeriod` (deliveryAt − 8 дней).
2. `submitOrderRating` / `skipOrderRating` → `rating_period_expired`.
3. `getCurrentOrder` → `null` для неоценённого expired заказа.
