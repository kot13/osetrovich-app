# Спецификация фичи: Корзина и оформление заказа

> **Язык**: этот документ и все артефакты Spec Kit MUST быть на **русском языке**
> (конституция проекта, принцип VII).

**Feature Branch**: `005-cart-checkout`

**Created**: 2026-07-14

**Status**: Implemented

**Input**: User description: "добавляем корзину и чекаут. Если корзина не пуста, то в ней
выводится: список покупок, сумма, поле ввода адреса, условия оплаты и доставки, поле
комментария к заказу, кнопка «Оформить». После оформления — сообщение об успешном
оформлении. Корзина очищается."

## User Scenarios & Testing *(mandatory)*

### User Story 1 — Просмотр непустой корзины (Priority: P1)

Покупатель открывает вкладку «Корзина» с одним или несколькими товарами и видит полноценный
экран оформления: список добавленных позиций, итоговую сумму с учётом доставки, поле адреса
доставки, блок условий оплаты и доставки, необязательное поле комментария и кнопку
«Оформить».

**Why this priority**: без содержимого корзины пользователь не может проверить заказ перед
оформлением; это базовый сценарий завершения покупки.

**Independent Test**: добавить товары из каталога, открыть «Корзину» и убедиться, что
отображаются все обязательные блоки экрана и корректная сумма.

**Acceptance Scenarios**:

1. **Given** в корзине есть хотя бы одна позиция, **When** пользователь открывает вкладку
   «Корзина», **Then** отображается список покупок (каждая позиция с названием, весом,
   ценой за единицу, количеством и суммой по строке); пустое состояние «В корзине пока
   пусто» не показывается.
2. **Given** в корзине несколько позиций, **When** пользователь просматривает экран,
   **Then** отображается итоговая сумма заказа с разбивкой: стоимость товаров, стоимость
   доставки (если применима) и итого к оплате.
3. **Given** корзина не пуста, **When** пользователь смотрит на экран, **Then** видны поле
   ввода адреса доставки, блок условий оплаты и доставки (текст согласно FR-005), поле
   комментария к заказу и кнопка «Оформить».
4. **Given** сумма товаров в корзине **от 2000 руб. включительно**, **When** отображается
   итог, **Then** стоимость доставки по Санкт-Петербургу и ближайшему пригороду равна
   **0 руб.** (бесплатно).
5. **Given** сумма товаров в корзине **меньше 2000 руб.**, **When** отображается итог,
   **Then** стоимость доставки по Санкт-Петербургу и ближайшему пригороду равна
   **300 руб.**

### User Story 2 — Изменение состава корзины на экране (Priority: P1)

Покупатель может скорректировать заказ прямо на экране «Корзина»: увеличить или уменьшить
количество позиции либо удалить её. Список, суммы и бейдж на вкладке «Корзина» в Tab Bar
обновляются **реактивно**.

**Why this priority**: корректировка состава перед оплатой — обязательная часть UX корзины;
без неё пользователь вынужден возвращаться в каталог.

**Independent Test**: на экране корзины изменить количество и удалить позицию; проверить
пересчёт суммы и значение бейджа на Tab Bar.

**Acceptance Scenarios**:

1. **Given** позиция в корзине с количеством K > 1, **When** пользователь уменьшает
   количество, **Then** количество и сумма по строке уменьшаются; итоговая сумма заказа
   пересчитывается.
2. **Given** позиция в корзине с количеством 1, **When** пользователь уменьшает
   количество до нуля, **Then** позиция удаляется из списка; бейдж Tab Bar показывает
   число **оставшихся различных** товарных позиций.
3. **Given** позиция в корзине, **When** пользователь увеличивает количество,
   **Then** количество и суммы обновляются без перезагрузки экрана.
4. **Given** после удаления последней позиции корзина пуста, **When** экран обновляется,
   **Then** отображается пустое состояние «В корзине пока пусто» с кнопкой «Перейти в
   каталог» (поведение из `001-init-app-shell`).

### User Story 3 — Оформление заказа (Priority: P1)

Авторизованный покупатель заполняет адрес доставки (и при необходимости комментарий),
нажимает «Оформить» и получает подтверждение успешного оформления. Корзина очищается;
бейдж на вкладке «Корзина» исчезает.

**Why this priority**: оформление заказа — целевое действие всей корзины; без него фича не
приносит бизнес-ценности.

**Independent Test**: с непустой корзиной и заполненным адресом нажать «Оформить»;
убедиться в сообщении об успехе и пустой корзине.

**Acceptance Scenarios**:

1. **Given** пользователь авторизован, корзина не пуста и адрес доставки заполнен,
   **When** пользователь нажимает «Оформить», **Then** заказ принимается; показывается
   сообщение об успешном оформлении; корзина очищается; бейдж Tab Bar скрывается.
2. **Given** пользователь авторизован, корзина не пуста, поле адреса пустое или содержит
   только пробелы, **When** пользователь нажимает «Оформить», **Then** заказ **не**
   оформляется; пользователь видит понятное сообщение о необходимости указать адрес;
   содержимое корзины сохраняется.
3. **Given** пользователь **не** авторизован и корзина не пуста, **When** он нажимает
   «Оформить», **Then** оформление не выполняется; пользователю предлагается войти в
   аккаунт (переход к сценарию авторизации из `001-init-app-shell`).
4. **Given** неавторизованный пользователь нажал «Оформить» с заполненным адресом,
   **When** он успешно проходит авторизацию из сценария checkout (`from=checkout`),
   **Then** приложение возвращает его на экран «Корзина» и **автоматически** оформляет
   заказ с сохранёнными адресом и комментарием; показывается сообщение об успехе;
   корзина очищается.
5. **Given** заказ успешно оформлен, **When** пользователь закрывает сообщение об успехе
   или остаётся на экране «Корзина», **Then** отображается пустое состояние корзины.
6. **Given** пользователь оформляет заказ с комментарием, **When** заказ принят,
   **Then** комментарий передаётся вместе с заказом (отображается в подтверждении только
   косвенно — через уведомление с деталями заказа согласно условиям доставки).

### Edge Cases

- Что происходит, если при оформлении заказа возникает ошибка (сеть, сервер)? Пользователь
  видит сообщение об ошибке; корзина **не** очищается; введённые адрес и комментарий
  сохраняются на экране.
- Что происходит при сумме товаров ровно 2000 руб.? Доставка бесплатна (порог «от 2000 руб.»
  включительно).
- Что происходит, если пользователь меняет состав корзины так, что сумма переходит порог
  2000 руб.? Стоимость доставки и итог пересчитываются немедленно.
- Что происходит при повторном нажатии «Оформить» во время обработки? Повторная отправка
  блокируется до завершения текущей попытки (защита от дублирования заказа).
- Что происходит, если товар из корзины стал недоступен на сервере при оформлении?
  Пользователь получает понятное сообщение; недоступная позиция удаляется или помечается;
  оформление не завершается до исправления состава.
- Что происходит, если пользователь прервал авторизацию при оформлении (закрыл экран
  входа)? Заказ не создаётся; адрес и комментарий остаются в полях формы; отложенное
  оформление (`PendingCheckout`) сбрасывается только после успешного заказа.
- Что происходит, если сессия восстановлена из хранилища, но профиль в мок-API не был
  загружен? Перед `POST /orders` клиент синхронизирует профиль из JWT (см. research §6).

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: Экран «Корзина» MUST заменять пустое состояние из `001-init-app-shell` на
  полноценный экран оформления, когда в корзине есть хотя бы одна позиция.
- **FR-002**: Список покупок MUST показывать для каждой позиции: название товара, вес,
  цену за единицу, количество, сумму по строке и элементы изменения количества
  («−» / «+») с удалением при уменьшении до нуля.
- **FR-003**: Экран MUST отображать итоговую сумму заказа: стоимость товаров, стоимость
  доставки и **итого к оплате**; суммы MUST пересчитываться реактивно при изменении
  состава корзины.
- **FR-004**: Стоимость доставки MUST рассчитываться по правилам:
  - сумма товаров **≥ 2000 руб.** → доставка **0 руб.** (бесплатно);
  - сумма товаров **< 2000 руб.** → доставка **300 руб.**
  для доставки по Санкт-Петербургу и ближайшему пригороду.
- **FR-005**: Экран MUST содержать информационный блок условий оплаты и доставки с текстом:

  > После формирования заказа поступит сообщение с его деталями и окончательной суммой.
  > Доставка от 2000 руб. осуществляется бесплатно по Санкт-Петербургу и ближайшему пригороду.
  > Доставка до 2000 руб. по Санкт-Петербургу и ближайшему пригороду 300 руб.
  > Оплата при получении наличными курьеру или банковской картой.

- **FR-006**: Экран MUST содержать обязательное для оформления поле **адреса доставки**
  (многострочное или однострочное — на усмотрение дизайна; минимум — свободный ввод
  текста адреса).
- **FR-007**: Экран MUST содержать **необязательное** поле комментария к заказу.
- **FR-008**: Кнопка «Оформить» MUST быть доступна только при непустой корзине;
  оформление MUST требовать заполненного адреса и активной авторизованной сессии
  пользователя.
- **FR-009**: При успешном оформлении система MUST показать сообщение об успешном
  оформлении заказа, MUST очистить корзину и MUST обновить бейдж вкладки «Корзина»
  (скрыть при нуле позиций).
- **FR-010**: При пустой корзине экран MUST сохранять поведение `001-init-app-shell`:
  текст «В корзине пока пусто» и кнопка «Перейти в каталог».
- **FR-011**: Состояние корзины (позиции и количества) MUST оставаться синхронизированным
  с каталогом и страницей товара (фича `004-product-catalog`) и с бейджем Tab Bar.
- **FR-012**: Неавторизованный пользователь MUST иметь возможность наполнять корзину и
  просматривать экран «Корзина»; оформление заказа MUST быть доступно только после входа
  в аккаунт.
- **FR-013**: При переходе на авторизацию из checkout клиент MUST сохранить адрес и
  комментарий в `PendingCheckout`; после успешного входа с `from=checkout` MUST вернуть
  пользователя на `/cart` и автоматически отправить заказ без повторного нажатия «Оформить».

### Key Entities

- **Позиция корзины (Cart Line Item)**: товар из каталога с количеством; атрибуты для
  отображения — название, вес, цена за единицу, количество, сумма по строке.
- **Заказ (Order)**: оформленная покупка; включает список позиций, адрес доставки,
  необязательный комментарий, стоимость товаров, стоимость доставки, итоговую сумму,
  идентификатор/номер заказа (для уведомления пользователю).
- **Адрес доставки**: текстовое значение, введённое пользователем на экране корзины.
- **Корзина (Cart)**: совокупность позиций до оформления; после успешного оформления
  MUST быть пустой.
- **Отложенное оформление (PendingCheckout)**: сохранённые адрес и комментарий при
  переходе на авторизацию из checkout; очищается после успешного заказа.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: Пользователь с непустой корзиной видит полный экран оформления (список,
  суммы, адрес, условия, комментарий, кнопку) **с первого открытия** вкладки «Корзина»
  без дополнительных действий.
- **SC-002**: **95%** пользователей успешно оформляют заказ с первой попытки при
  заполненном адресе и стабильном соединении (измеряется на этапе приёмочного тестирования).
- **SC-003**: Пересчёт итоговой суммы при изменении количества или удалении позиции
  происходит **без заметной задержки** для пользователя (мгновенно на клиенте).
- **SC-004**: После успешного оформления корзина пуста в **100%** случаев; бейдж Tab Bar
  не показывает устаревшее количество позиций.
- **SC-005**: Оформление заказа от заполнения адреса до сообщения об успехе занимает
  **не более 10 секунд** при нормальных условиях сети.

## Assumptions

- Наполнение корзины, бейдж Tab Bar и синхронизация с каталогом уже реализованы в фиче
  `004-product-catalog`; данная фича добавляет экран непустой корзины и оформление заказа.
- Пустое состояние корзины и навигация «Перейти в каталог» наследуются из
  `001-init-app-shell` без изменений текстов.
- Оформление заказа требует **авторизации** (телефон из профиля нужен для уведомления
  «поступит сообщение с деталями заказа»); просмотр и редактирование корзины — без входа.
- Адрес доставки вводится вручную на экране корзины; автоподстановка из профиля **вне
  scope** данной фичи (может быть добавлена позже).
- Зона доставки — Санкт-Петербург и ближайший пригород; проверка географической
  принадлежности адреса **не** выполняется — пользователь самостоятельно указывает адрес
  в зоне доставки.
- Оплата при получении (наличные или карта курьеру); онлайн-оплата в приложении **вне
  scope**.
- Уведомление с деталями заказа отправляется внешней системой (СМС/push) после приёма
  заказа; содержимое и канал уведомления определяются бэкендом.
- Форматирование цен — в рублях, как в каталоге (фича `004-product-catalog`).
- История заказов и экран «Мои заказы» **вне scope** данной фичи.
