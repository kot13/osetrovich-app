# Quickstart: Оформление заказа через API

**Фича**: [spec.md](./spec.md) | **План**: [plan.md](./plan.md) | **Модель**: [data-model.md](./data-model.md)

Руководство по проверке фичи после реализации.

## Предварительные требования

- Flutter SDK (stable), Dart 3.x
- Репозиторий с выполненными задачами из `tasks.md`
- Мок-API включён (`useMockApi = true` в `lib/core/network/providers.dart`)
- OpenAPI v0.11.2 в `openapi/openapi.yaml` (схемы orders с `id: integer`, `apartment`)
- Для оформления — **авторизованная** сессия (СМС-вход)
- Корзина и базовый checkout реализованы (`005-cart-checkout`)

## Установка и запуск

```bash
flutter pub get
flutter run
```

## Сценарии ручной проверки

### С1. Поле «Квартира / офис» (US2, FR-004)

1. Добавить товары, открыть «Корзину».
2. Убедиться: между адресом и комментарием есть поле «Квартира / офис» с подсказкой
   «Необязательно».
3. Заполнить только адрес, оставить квартиру пустой — «Оформить» доступно.
4. Оформить заказ — успех.

---

### С2. Заказ с квартирой (US1, FR-002, FR-003)

1. Авторизоваться, наполнить корзину.
2. Заполнить адрес и квартиру (например, «42»), нажать «Оформить».
3. Убедиться в диалоге успеха; корзина пуста.
4. (Опционально) проверить в моке / логах, что в запросе есть `apartment: "42"`.

---

### С3. Целочисленные id товаров (US1, FR-001)

1. Добавить 2 разных товара с int id из каталога.
2. Оформить заказ с заполненным адресом.
3. Убедиться в успешном оформлении (мок принимает `items[].id` как integer).

---

### С4. Снятие фокуса (US3, FR-009, FR-010)

1. Открыть корзину, тапнуть поле адреса — клавиатура открылась.
2. Тапнуть по строке товара или блоку условий доставки (вне поля ввода).
3. Клавиатура скрылась, фокус снят; введённый текст в поле сохранён.
4. Нажать «Оформить» — кнопка срабатывает штатно.
5. Тапнуть «+» на количестве — количество меняется.

---

### С5. Отложенное оформление с квартирой (FR-008)

1. Выйти из аккаунта, наполнить корзину.
2. Заполнить адрес, квартиру и комментарий.
3. Нажать «Оформить» → пройти авторизацию (`from=checkout`).
4. После входа — автоматическое оформление с сохранёнными полями; успех.

---

### С6. Ошибка оформления — данные формы (FR-007, SC-004)

1. Симулировать ошибку API (недоступный товар / сетевой сбой).
2. Убедиться: адрес, квартира и комментарий остались в полях; корзина не очищена.

---

### С7. Регрессия базового checkout (FR-006, FR-011)

1. Проверить расчёт доставки (0 ₽ от 2000 ₽, иначе 300 ₽).
2. Пустой адрес → сообщение «Укажите адрес доставки».
3. Неавторизованный пользователь → переход на вход.
4. Успех → пустая корзина, бейдж Tab Bar скрыт.
