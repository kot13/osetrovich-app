## User Scenarios & Testing *(mandatory)*

### User Story 1 — Бейдж «Товар недели» на карточках товара (Priority: P1)

Покупатель видит на карточке товара визуальный бейдж **«Товар недели»**, если товар
помечен соответствующим признаком в данных каталога. Бейдж отображается вместе с уже
существующими бейджами «Акция» и «СПЕЦЦЕНА» по тем же правилам компоновки: заметен на
фоне фото, не перекрывает название, вес и цену.

**Why this priority**: признак пришёл из обновлённого контракта API; без отображения
покупатель не видит выделение товаров недели, а блок «Товары недели» на главной теряет
смысл как промо-инструмент.

**Independent Test**: открыть каталог или ленту «Товары недели» с товаром, у которого
признак «Товар недели» включён, и убедиться, что бейдж отображается; для товара без
признака бейдж отсутствует.

**Acceptance Scenarios**:

1. **Given** товар с признаком «Товар недели» в сетке каталога,
   **When** пользователь смотрит на карточку,
   **Then** на карточке виден бейдж с текстом «Товар недели».
2. **Given** товар без признака «Товар недели»,
   **When** пользователь смотрит на карточку,
   **Then** бейдж «Товар недели» не отображается.
3. **Given** товар с признаком «Товар недели» в горизонтальной ленте «Товары недели» на
   главной,
   **When** пользователь смотрит на карточку,
   **Then** бейдж отображается так же, как в каталоге.
4. **Given** товар одновременно с признаками «Товар недели», «Акция» и «СПЕЦЦЕНА»,
   **When** пользователь смотрит на карточку,
   **Then** отображаются все применимые бейджи без наложения друг на друга.
5. **Given** товар с признаком «Товар недели» на странице детали товара,
   **When** пользователь просматривает экран,
   **Then** бейдж «Товар недели» виден на экране детали (если на детали уже показываются
   промо-бейджи) либо признак учтён в данных без ошибок загрузки.

### User Story 2 — Переход по диплинку из статьи (Priority: P1)

Покупатель читает акцию или новость в разделе «Акции и Новости». В тексте материала есть
ссылка формата `osetrovich://…` (например, на товар, категорию или другую статью).
Нажатие на такую ссылку открывает соответствующий экран **внутри приложения**, без
перехода во внешний браузер.

**Why this priority**: контент-менеджеры смогут вести пользователя из статей на целевые
разделы магазина; это основной канал маркетинговых переходов помимо push.

**Independent Test**: открыть статью с встроенной ссылкой `osetrovich://catalog/product/{id}`,
нажать на ссылку и убедиться, что открывается страница указанного товара.

**Acceptance Scenarios**:

1. **Given** пользователь на экране статьи, **When** в тексте нажата ссылка
   `osetrovich://catalog/product/{id}` с существующим товаром,
   **Then** открывается страница этого товара.
2. **Given** пользователь на экране статьи, **When** нажата ссылка
   `osetrovich://catalog/category/{id}` с существующей категорией,
   **Then** открывается вкладка «Каталог» с выбранной категорией.
3. **Given** пользователь на экране статьи, **When** нажата ссылка
   `osetrovich://promotions/articles/{id}` с существующим материалом,
   **Then** открывается детальная страница этой акции/новости.
4. **Given** пользователь на экране статьи, **When** нажата ссылка `osetrovich://home`,
   **Then** открывается вкладка «Главная».
5. **Given** пользователь на экране статьи, **When** нажата ссылка на раздел без
   идентификатора (`osetrovich://catalog`, `osetrovich://promotions`, `osetrovich://profile`,
   `osetrovich://notifications`),
   **Then** открывается соответствующий корневой экран раздела; для каталога активна
   вкладка «Все».
6. **Given** в статье есть обычная внешняя ссылка (`https://…`),
   **When** пользователь нажимает на неё,
   **Then** ссылка открывается во внешнем браузере, как и раньше.

### User Story 3 — Переход по диплинку из push-уведомления (Priority: P1)

Покупатель получает push-уведомление, в данных которого указана ссылка формата
`osetrovich://…`. При нажатии на уведомление приложение открывается (или выводится на
передний план) и переходит на целевой экран согласно ссылке.

**Why this priority**: push — ключевой канал возврата пользователей; единая схема ссылок
для статей и push упрощает настройку рассылок.

**Independent Test**: симулировать push с полем ссылки `osetrovich://notifications/{id}`,
нажать на уведомление и убедиться, что открывается детальная страница уведомления.

**Acceptance Scenarios**:

1. **Given** приложение закрыто, **When** пользователь нажимает push со ссылкой
   `osetrovich://home`,
   **Then** приложение запускается на вкладке «Главная».
2. **Given** приложение в фоне, **When** пользователь нажимает push со ссылкой
   `osetrovich://catalog/product/{id}`,
   **Then** открывается страница указанного товара.
3. **Given** приложение активно, **When** пользователь нажимает push со ссылкой
   `osetrovich://notifications`,
   **Then** открывается список уведомлений.
4. **Given** push содержит ссылку `osetrovich://notifications/{id}` с существующим
   уведомлением,
   **When** пользователь нажимает на уведомление,
   **Then** открывается детальная страница этого уведомления.
5. **Given** push содержит ссылку `osetrovich://profile`,
   **When** пользователь нажимает на уведомление,
   **Then** открывается вкладка «Профиль».
6. **Given** push не содержит ссылки `osetrovich://` (пустой payload или только
   title/body),
   **When** пользователь нажимает на уведомление,
   **Then** сохраняется текущее поведение: переход на список уведомлений.

### User Story 4 — Открытие диплинка извне приложения (Priority: P2)

Покупатель переходит по ссылке `osetrovich://…` из внешнего контекста (например, из
другого приложения или системного обработчика ссылок). Приложение открывается на нужном
экране по той же таблице маршрутов, что и для статей и push.

**Why this priority**: единая схема ссылок должна работать независимо от источника;
поддержка внешнего открытия закрепляет контракт для будущих интеграций.

**Independent Test**: открыть ссылку `osetrovich://promotions` через системный механизм
(adb/xcrun или тестовый intent) и убедиться, что приложение показывает раздел «Акции и
Новости».

**Acceptance Scenarios**:

1. **Given** приложение не запущено, **When** ОС обрабатывает ссылку `osetrovich://catalog`,
   **Then** приложение стартует на вкладке «Каталог» с фильтром «Все».
2. **Given** приложение запущено на другой вкладке, **When** ОС обрабатывает ссылку
   `osetrovich://promotions/articles/{id}`,
   **Then** открывается детальная страница указанного материала.

---

### Edge Cases

- Несуществующий `id` в диплинке (товар, категория, статья, уведомление): показывается
  понятное пустое состояние или сообщение об ошибке на целевом экране; приложение не
  завершается аварийно.
- Некорректный или неизвестный путь диплинка (например, `osetrovich://unknown`): переход
  на «Главную» без краша.
- Диплинк с нечисловым `id` там, где ожидается число (категория, товар): обрабатывается
  как некорректная ссылка с переходом на «Главную».
- Диплинк на уведомления при неавторизованном пользователе: открывается экран входа или
  список уведомлений с штатным поведением для гостя — согласно существующим правилам
  раздела уведомлений.
- Одновременное наличие в push JSON-payload (формат фичи 008) и URL `osetrovich://`: URL
  `osetrovich://` имеет приоритет, если присутствует; иначе используется JSON-payload;
  если оба отсутствуют — переход на список уведомлений.
- Повторное нажатие на ту же ссылку, когда пользователь уже на целевом экране: экран
  остаётся стабильным, без дублирования стека навигации.
- Ссылка `osetrovich://catalog` при отсутствии сети: открывается каталог с уже известным
  состоянием или штатным сообщением об ошибке загрузки.

## Requirements *(mandatory)*

### Functional Requirements

**Признак «Товар недели»**

- **FR-001**: Приложение MUST получать и хранить булев признак «Товар недели» для каждого
  товара согласно обновлённому контракту API (`productOfWeek`).
- **FR-002**: На карточке товара в каталоге и в ленте «Товары недели» MUST отображаться
  бейдж «Товар недели», если признак включён.
- **FR-003**: Бейдж «Товар недели» MUST сосуществовать с бейджами «Акция» и «СПЕЦЦЕНА» по
  тем же визуальным правилам, что и существующие промо-бейджи.
- **FR-004**: Моки данных MUST возвращать признак `productOfWeek` в соответствии с
  OpenAPI; часть товаров в моках MUST иметь `productOfWeek: true` для тестирования.

**Диплинки — схема и маршруты**

- **FR-005**: Приложение MUST регистрировать пользовательскую URL-схему `osetrovich` на
  Android и iOS.
- **FR-006**: Приложение MUST обрабатывать следующие пути и открывать соответствующие
  экраны:

  | Диплинк | Целевой экран |
  |---------|---------------|
  | `osetrovich://home` | Вкладка «Главная» |
  | `osetrovich://catalog` | Вкладка «Каталог», фильтр «Все» |
  | `osetrovich://catalog/category/{id}` | Вкладка «Каталог», выбранная категория `{id}` |
  | `osetrovich://catalog/product/{id}` | Страница товара `{id}` |
  | `osetrovich://promotions` | Раздел «Акции и Новости» |
  | `osetrovich://promotions/articles/{id}` | Детальная страница акции/новости `{id}` |
  | `osetrovich://profile` | Вкладка «Профиль» |
  | `osetrovich://notifications` | Список уведомлений |
  | `osetrovich://notifications/{id}` | Детальная страница уведомления `{id}` |

- **FR-007**: Ссылки `osetrovich://…` в HTML-теле статей (акции/новости) MUST открываться
  внутри приложения, а не во внешнем браузере.
- **FR-008**: Push-уведомления MUST поддерживать навигацию по ссылкам `osetrovich://…` при
  нажатии на уведомление (холодный старт, фон, активное приложение).
- **FR-009**: При некорректном или неизвестном диплинке приложение MUST перенаправлять
  пользователя на «Главную» без аварийного завершения.
- **FR-010**: При диплинке на сущность с несуществующим `id` целевой экран MUST показать
  штатное пустое состояние или сообщение об ошибке.
- **FR-011**: Существующий механизм push-навигации через JSON-payload (фича 008) MUST
  сохраняться; при наличии URL `osetrovich://` в данных push URL имеет приоритет.

### Key Entities

- **Товар (ProductSummary / ProductDetail)**: атрибут `productOfWeek` (boolean) — признак
  «Товар недели» из ERP; влияет на отображение бейджа и участие в блоке «Товары недели»
  (порядок в списке определяется сервером).
- **Диплинк (Deep Link)**: URL вида `osetrovich://<path>` с опциональным числовым или
  строковым идентификатором; источники — HTML статей, push-данные, внешние intent/universal
  links.
- **Маршрут навигации**: соответствие между путём диплинка и экраном приложения; единый
  для всех источников ссылок.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 100% перечисленных в FR-006 диплинков открывают корректный экран при
  тестировании на Android и iOS (9 маршрутов × 2 платформы).
- **SC-002**: Пользователь переходит из статьи на целевой экран по внутренней ссылке за
  одно нажатие, без промежуточного открытия браузера.
- **SC-003**: Push с валидным диплинком доставляет пользователя на целевой экран в течение
  3 секунд после нажатия на уведомление (при нормальной загрузке приложения).
- **SC-004**: Товары с признаком «Товар недели» визуально отличимы от остальных в каталоге
  и на главной — бейдж заметен при беглом просмотре ленты.
- **SC-005**: Некорректные и битые диплинки не приводят к падению приложения в 100%
  проверенных сценариев (минимум: неизвестный путь, несуществующий id, невалидный id).

## Assumptions

- OpenAPI в репозитории уже содержит поле `productOfWeek` в схемах `ProductSummary` и
  `ProductDetail`; задача фичи — синхронизировать клиент и моки, а не менять контракт.
- Текст бейджа — **«Товар недели»**; визуальный стиль согласуется с существующими
  промо-бейджами (отдельный цвет может быть определён на этапе планирования).
- Идентификаторы категорий и товаров в диплинках — целые числа; идентификаторы статей и
  уведомлений — строки, как в текущем API.
- Push-рассылки могут передавать URL `osetrovich://…` в поле данных уведомления (например,
  `url` или `deeplink`); точное имя поля определяется на этапе планирования совместно с
  контрактом push-deeplink.
- Universal Links / App Links для домена `osetrovich.ru` не входят в scope; поддерживается
  только схема `osetrovich://`.
- Раздел уведомлений по-прежнему доступен с вкладки «Главная»; диплинк `osetrovich://notifications`
  открывает тот же список, что и маршрут `/home/notifications`.
- Диплинк `osetrovich://promotions/articles/{id}` соответствует экрану детали материала;
  путь внутренней навигации может отличаться (`/promotions/article/{id}`), но пользовательский
  результат идентичен.
