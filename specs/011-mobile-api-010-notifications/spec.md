# Спецификация фичи: Mobile API 0.10.0 — push-токены и реальные уведомления

> **Язык**: этот документ и все артефакты Spec Kit MUST быть на **русском языке**
> (конституция проекта, принцип VII).

**Feature Branch**: `011-mobile-api-010-notifications`

**Created**: 2026-07-20

**Status**: Draft

**Input**: User description: «Переход Mobile API 0.9.0 → 0.10.0. Формально в OpenAPI добавлен
один эндпоинт, но поведение уведомлений изменилось существенно — это главное для приложения.
Обязательна регистрация FCM-токена (PUT /v1/profile/push-token), переход с stub-данных на реальные
уведомления (list / unread-count / read / read-all), синхронизация pushEnabled с настройками,
обработка push в foreground/background, новые типы уведомлений по title/body, сценарий «Заказ
доставлен» → оценка заказа.»

## User Scenarios & Testing *(mandatory)*

### User Story 1 — Регистрация push-токена для доставки уведомлений (Priority: P1)

Авторизованный покупатель входит в приложение или включает push-уведомления в настройках.
Приложение запрашивает разрешение операционной системы (если ещё не выдано), получает
регистрационный токен устройства и отправляет его на сервер вместе с платформой (iOS или
Android). Без успешной регистрации токена сервер не сможет доставить push, даже если в
настройках профиля включены уведомления.

**Why this priority**: без регистрации push-токена вся цепочка уведомлений не работает — это
блокирующее требование API 0.10.0.

**Independent Test**: авторизоваться, убедиться что токен зарегистрирован на сервере (ответ 204),
затем отправить тестовое push с бэкенда и получить его на устройстве.

**Acceptance Scenarios**:

1. **Given** пользователь успешно прошёл вход по SMS, **When** сессия установлена и получен
   регистрационный токен устройства, **Then** приложение отправляет токен на сервер
   (`PUT /v1/profile/push-token`) с полями `token` и `platform` (`ios` или `android`) и
   получает ответ 204.
2. **Given** пользователь авторизован, **When** платформа обновляет регистрационный токен
   (событие обновления токена), **Then** приложение повторно регистрирует новый токен на сервере.
3. **Given** в настройках профиля push-уведомления выключены, **When** пользователь включает
   переключатель «Уведомления» (`pushEnabled: true` через `PATCH /v1/profile/preferences`),
   **Then** приложение запрашивает разрешение ОС (если нужно) и регистрирует push-токен на сервере.
4. **Given** регистрация токена выполняется с пустым значением `token`, **When** сервер отвечает
   422, **Then** приложение не считает регистрацию успешной и не блокирует работу остальных функций.
5. **Given** сессия недействительна (ответ 401 на регистрацию токена), **When** приложение
   обрабатывает ошибку, **Then** пользователь переводится в состояние повторного входа согласно
   существующей логике сессии.

### User Story 2 — Реальный список уведомлений вместо заглушек (Priority: P1)

Авторизованный покупатель открывает раздел «Уведомления» с главного экрана (колокольчик) и видит
актуальный список сообщений с сервера: идентификатор, заголовок, текст, дату и статус прочитанности.
Счётчик непрочитанных на колокольчике отражает данные сервера, а не локальные mock-значения.

**Why this priority**: пользователь должен видеть реальные сообщения о заказах и статусах;
stub-данные (`n1`, `n2` и фиксированные списки) больше не соответствуют контракту.

**Independent Test**: авторизоваться, дождаться загрузки уведомлений с сервера, проверить список,
счётчик непрочитанных, отметку одного уведомления прочитанным и «Отметить все прочитанным».

**Acceptance Scenarios**:

1. **Given** пользователь авторизован, **When** приложение завершает вход, **Then** загружается
   список уведомлений (`GET /v1/notifications`) и счётчик непрочитанных
   (`GET /v1/notifications/unread-count`).
2. **Given** на сервере есть уведомления с реальными идентификаторами (например, `"42"`),
   **When** пользователь открывает раздел «Уведомления», **Then** отображаются заголовок, текст,
   дата (`createdAt`) и визуальное различие прочитанных / непрочитанных; stub-идентификаторы
   (`n1`, `n2`) и локальные mock-списки не используются.
3. **Given** пользователь открывает непрочитанное уведомление, **When** отображается экран
   деталей, **Then** приложение вызывает `POST /v1/notifications/{id}/read`, уведомление
   помечается прочитанным, счётчик на колокольчике уменьшается.
4. **Given** есть непрочитанные уведомления, **When** пользователь нажимает «Отметить все
   прочитанным», **Then** приложение вызывает `POST /v1/notifications/read-all`, список и
   счётчик обновляются.
5. **Given** пользователь пытается отметить прочитанным несуществующее или чужое уведомление,
   **When** сервер отвечает 404, **Then** приложение корректно обрабатывает ошибку (без сбоя
   интерфейса) и при необходимости обновляет список с сервера.
6. **Given** пользователь обновил приложение до версии с API 0.10.0, **When** приложение
   запускается впервые после обновления, **Then** локальный кэш уведомлений из предыдущей версии
   очищается и данные загружаются заново с сервера.

### User Story 3 — Управление push-уведомлениями в настройках (Priority: P2)

Покупатель включает или выключает push-уведомления переключателем в профиле. Состояние
синхронизируется с сервером (`pushEnabled` в `PATCH /v1/profile/preferences`). Приложение не
выбирает канал доставки (push vs SMS) — это решает сервер; приложение обеспечивает корректную
регистрацию токена при включении.

**Why this priority**: без синхронизации настроек пользователь не получит ожидаемое поведение
уведомлений, даже при зарегистрированном токене.

**Independent Test**: переключить «Уведомления» в профиле, проверить что значение ушло на сервер,
при включении — зарегистрирован токен; при выключении — push не ожидается (сервер переключится
на SMS самостоятельно).

**Acceptance Scenarios**:

1. **Given** пользователь в настройках профиля, **When** он переключает «Уведомления»,
   **Then** приложение отправляет `PATCH /v1/profile/preferences` с актуальным `pushEnabled`.
2. **Given** пользователь включает уведомления, **When** разрешение ОС ещё не выдано,
   **Then** приложение запрашивает разрешение; при отказе — информирует пользователя, что push
   недоступны, без падения приложения.
3. **Given** пользователь выключает уведомления, **When** переключатель в положении «выкл»,
   **Then** приложение может удалить локально сохранённый токен; сервер самостоятельно
   переключится на SMS при необходимости.
4. **Given** `pushEnabled = true`, но push не доставился (нет токена, ошибка доставки),
   **When** пользователь открывает приложение, **Then** уведомление доступно в in-app списке;
   SMS-fallback в этом случае не выполняется (поведение сервера).

### User Story 4 — Получение и открытие push-уведомлений (Priority: P2)

Покупатель получает push-сообщение от магазина (заголовок и текст в payload). В фоне — системное
уведомление в трее; в активном приложении — in-app баннер или обновление списка и счётчика. По
нажатию на push открывается экран уведомлений.

**Why this priority**: завершает пользовательский цикл доставки уведомлений после регистрации
токена.

**Independent Test**: отправить push на зарегистрированное устройство, проверить отображение в
фоне и foreground, нажать на уведомление и убедиться в переходе на экран уведомлений.

**Acceptance Scenarios**:

1. **Given** приложение в фоне, **When** приходит push с `title` и `body`, **Then** отображается
   системное уведомление в трее операционной системы.
2. **Given** приложение открыто (foreground), **When** приходит push, **Then** пользователь
   видит in-app баннер или обновлённые список уведомлений и счётчик непрочитанных.
3. **Given** пользователь нажимает на push-уведомление, **When** приложение открывается или
   выходит на передний план, **Then** открывается экран «Уведомления» (в API нет поля `type` для
   deep link — навигация по заголовку не требуется).
4. **Given** после нажатия на push известен идентификатор уведомления (из in-app списка),
   **When** пользователь просматривает уведомление, **Then** приложение может отметить его
   прочитанным через `POST /v1/notifications/{id}/read`.

### User Story 5 — Уведомление «Заказ доставлен» и призыв к оценке (Priority: P3)

Покупатель получает уведомление с заголовком «Заказ доставлен». На экране деталей или в списке
отображается призыв оставить отзыв. Приложение не извлекает номер заказа из текста уведомления,
а использует данные текущего заказа с сервера для перехода к оценке.

**Why this priority**: улучшает конверсию в отзывы и соответствует новому формату текстов
уведомлений API 0.10.0.

**Independent Test**: получить или отобразить уведомление «Заказ доставлен», нажать CTA оценки,
убедиться что открывается экран оценки при `ratingState: pending` у текущего заказа.

**Acceptance Scenarios**:

1. **Given** уведомление с заголовком «Заказ доставлен», **When** пользователь открывает его,
   **Then** отображается текст с призывом оставить отзыв и кнопка/ссылка для оценки.
2. **Given** пользователь нажимает CTA оценки, **When** `GET /v1/orders/current` возвращает
   заказ с `ratingState: pending`, **Then** открывается экран оценки заказа
   (`POST /orders/{id}/rating`).
3. **Given** уведомление «Заказ на доставке» с многострочным текстом (состав и сумма),
   **When** пользователь просматривает детали, **Then** переносы строк (`\n`) отображаются
   корректно.
4. **Given** любое уведомление о статусе заказа, **When** приложение отображает содержимое,
   **Then** номер заказа не парсится из текста уведомления (в т.ч. для «Заказ принят»).

---

### Edge Cases

- Пользователь отклонил разрешение на push в ОС — переключатель в профиле может быть включён,
  но токен не регистрируется; in-app уведомления остаются доступны.
- Сеть недоступна при регистрации токена — повторная попытка при следующем подходящем событии
  (вход, обновление токена, включение push).
- Пустой список уведомлений после входа — отображается пустое состояние, счётчик = 0.
- Одновременное обновление счётчика из push (foreground) и открытие списка — счётчик остаётся
  согласованным с сервером.
- Уведомление удалено на сервере, но ещё есть в локальном списке — при попытке прочитать (404)
  элемент удаляется или список перезагружается.
- Пользователь никогда не запускал приложение после регистрации — сервер отправляет SMS
  (логика сервера; приложение не участвует в выборе канала).
- Обновление приложения — очистка устаревшего локального кэша уведомлений (stub-данные).

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: Приложение MUST обновить контракт Mobile API до версии **0.10.0** (OpenAPI в
  репозитории и соответствующие моки).
- **FR-002**: Приложение MUST реализовать регистрацию push-токена через `PUT /v1/profile/push-token`
  с телом `{ "token": "<registration-token>", "platform": "ios" | "android" }` и обработкой
  ответов: 204 — успех, 401 — недействительная сессия, 422 — пустой token.
- **FR-003**: Регистрация push-токена MUST выполняться после успешного входа (`/auth/sms/verify`),
  при обновлении регистрационного токена устройства и после включения push в настройках
  (`pushEnabled: true`).
- **FR-004**: Приложение MUST удалить все stub-данные уведомлений: захардкоженные идентификаторы
  (`n1`, `n2`), локальные mock-списки и предположения о фиксированном/пустом списке.
- **FR-005**: После входа приложение MUST загружать уведомления (`GET /v1/notifications`) и счётчик
  непрочитанных (`GET /v1/notifications/unread-count`).
- **FR-006**: Приложение MUST поддерживать отметку прочитанным: `POST /v1/notifications/{id}/read`
  и `POST /v1/notifications/read-all`, с обработкой 404 для несуществующих уведомлений.
- **FR-007**: Переключатель «Уведомления» в профиле MUST синхронизировать `pushEnabled` через
  `PATCH /v1/profile/preferences`; при включении — запрос разрешения ОС и регистрация токена.
- **FR-008**: Приложение MUST NOT выбирать канал доставки (push vs SMS) — это ответственность
  сервера.
- **FR-009**: Push в фоне MUST отображаться как системное уведомление; в foreground — in-app
  баннер и/или обновление списка и счётчика.
- **FR-010**: Нажатие на push MUST открывать экран «Уведомления».
- **FR-011**: Для уведомления «Заказ доставлен» MUST отображаться CTA оценки; переход к оценке
  MUST использовать `GET /v1/orders/current` (`ratingState: pending`) и `POST /orders/{id}/rating`,
  без парсинга номера заказа из текста уведомления.
- **FR-012**: Многострочный `body` (например, «Заказ на доставке») MUST отображаться с сохранением
  переносов строк.
- **FR-013**: При первом запуске после обновления до версии с API 0.10.0 локальный кэш
  уведомлений MUST быть очищен.
- **FR-014**: Схема `Notification` MUST соответствовать контракту: `id` (string), `title`, `body`,
  `createdAt` (ISO8601 UTC), `isRead` (boolean).

### Key Entities

- **PushTokenRegistration**: регистрационный токен устройства и платформа (`ios` / `android`),
  привязанные к профилю пользователя на сервере.
- **Notification**: сообщение пользователю с идентификатором из БД, заголовком, текстом, датой
  создания и флагом прочитанности.
- **NotificationPreferences**: настройка `pushEnabled` в профиле, определяющая желание пользователя
  получать push (канал доставки выбирает сервер).
- **UnreadCount**: количество непрочитанных уведомлений для бейджа на колокольчике.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 100% авторизованных пользователей с выданным разрешением ОС на push и
  `pushEnabled: true` имеют зарегистрированный токен на сервере в течение 10 секунд после входа
  или включения уведомлений.
- **SC-002**: Раздел «Уведомления» отображает только данные с сервера; в коде и UI отсутствуют
  stub-идентификаторы и фиксированные mock-списки.
- **SC-003**: Счётчик непрочитанных на колокольчике совпадает с `GET /v1/notifications/unread-count`
  в 95% случаев проверки (допустима кратковременная задержка до 3 секунд после действия
  пользователя).
- **SC-004**: Пользователь может отметить одно уведомление или все сразу прочитанными; интерфейс
  обновляется без перезапуска приложения.
- **SC-005**: Push-уведомления, отправленные сервером на зарегистрированное устройство,
  отображаются в трее (фон) или in-app (foreground) в течение 5 секунд после доставки FCM.
- **SC-006**: Для уведомления «Заказ доставлен» пользователь может перейти к оценке заказа в
  не более чем 2 нажатия от экрана деталей уведомления.
- **SC-007**: После обновления приложения пользователь видит актуальный список с сервера, без
  устаревших локальных данных из предыдущей версии.

## Assumptions

- Firebase Cloud Messaging (FCM) уже интегрирован или будет интегрирован в рамках этой фичи;
  регистрационный токен получается стандартным SDK.
- Существующий UI раздела «Уведомления» (колокольчик, список, детали, «Отметить все прочитанным»)
  из фичи 002 сохраняется; меняется только источник данных и добавляется регистрация push-токена.
- Эндпоинты уведомлений (`GET /v1/notifications`, unread-count, read, read-all) уже описаны в
  OpenAPI 0.9.0; в 0.10.0 меняется только поведение (реальные данные) и добавляется push-token.
- В push payload с сервера передаются только `title` и `body`; идентификатор уведомления в push
  сейчас не передаётся — отметка прочитанным после tap возможна только при наличии id в in-app
  списке.
- Логика выбора канала push/SMS полностью на сервере; приложение не показывает пользователю,
  какой канал был использован.
- Экран оценки заказа уже существует или будет реализован в рамках связанной фичи; данная фича
  добавляет только навигацию из уведомления «Заказ доставлен».
- JWT-авторизация и автоматическое обновление токена при 401 (фича 010) применяются ко всем
  защищённым эндпоинтам этой фичи.

## Out of Scope

- Парсинг типа уведомления из API (поля `type` нет).
- SMS-fallback и логика выбора канала на стороне клиента.
- Передача `id` уведомления в push payload (если бэкенд не добавит это позже).
- Изменение текстов уведомлений на сервере (только отображение присланных `title` / `body`).
