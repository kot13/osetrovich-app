# Спецификация фичи: Push в foreground и диплинки уведомлений

> **Язык**: этот документ и все артефакты Spec Kit MUST быть на **русском языке**
> (конституция проекта, принцип VII).

**Feature Branch**: `017-push-foreground-deeplinks`

**Created**: 2026-07-22

**Status**: Draft

**Input**: User description: «1) На Android/iOS в foreground системный баннер часто не показывается,
если клиент не обрабатывает onMessage / onMessageReceived сам. In-app список при этом работает —
уведомление в API есть. Хочется получать пуш уведомления, когда приложение запущено и обновлять
каунтер в колокольчике. 2) Теперь в пушах приходит диплинк, чтобы можно было открыть уведомление.
Структура FCM: notification (title, body) + data (deeplink, notification_id). При нажатии на push
— открыть деталь уведомления; при foreground — MAY in-app баннер; при 404 — «Уведомление не
найдено»; не парсить notification.body для навигации.»

## User Scenarios & Testing *(mandatory)*

### User Story 1 — Обновление счётчика при push в активном приложении (Priority: P1)

Авторизованный покупатель пользуется приложением (экран открыт, приложение на переднем плане).
Сервер отправляет push о новом in-app уведомлении. Покупатель видит, что счётчик непрочитанных на
колокольчике на «Главной» обновился без перезапуска приложения и без ручного обновления экрана.
Уведомление появляется в in-app списке при его открытии.

**Why this priority**: без обработки входящих push в foreground пользователь не узнаёт о новых
сообщениях, пока сам не откроет раздел «Уведомления»; колокольчик остаётся устаревшим, хотя
данные на сервере уже есть.

**Independent Test**: при открытом приложении отправить тестовый push с `notification_id`,
убедиться что счётчик на колокольчике увеличился (или совпал с серверным unread-count) в течение
нескольких секунд; открыть список уведомлений и найти новое сообщение.

**Acceptance Scenarios**:

1. **Given** пользователь авторизован и приложение на переднем плане, **When** приходит push с
   полями `data.notification_id` и `notification.title` / `notification.body`, **Then**
   приложение обрабатывает входящее сообщение и обновляет счётчик непрочитанных на колокольчике
   согласно данным сервера.
2. **Given** пользователь на любом экране приложения (foreground), **When** приходит push,
   **Then** счётчик на колокольчике обновляется реактивно на всех экранах, где отображается
   колокольчик, без перезагрузки приложения.
3. **Given** push доставлен в foreground, **When** пользователь открывает раздел «Уведомления»,
   **Then** новое уведомление присутствует в списке с корректными заголовком и текстом.
4. **Given** пользователь не авторизован, **When** приходит push, **Then** приложение не падает;
   счётчик колокольчика не отображается или остаётся в состоянии для неавторизованного пользователя
   (согласно существующему поведению главного экрана).
5. **Given** push-уведомления выключены в настройках профиля (`pushEnabled: false`), **When**
   push всё же доставлен на устройство, **Then** приложение MAY игнорировать обновление счётчика
   из push; in-app список остаётся доступен при ручном открытии раздела.

### User Story 2 — Переход в деталь уведомления по нажатию на push (Priority: P1)

Покупатель получает push о заказе или статусе. При нажатии на системное уведомление (приложение
в фоне или закрыто) приложение открывается и показывает экран детали **конкретного** уведомления,
идентификатор которого передан в `data.deeplink` или `data.notification_id`.

**Why this priority**: основная ценность push — мгновенный переход к релевантному сообщению; без
диплинка пользователь попадает только в общий список и теряет контекст.

**Independent Test**: отправить push с `data.deeplink: osetrovich://notifications/{id}` и
`data.notification_id: {id}`, нажать на системное уведомление при закрытом приложении и убедиться,
что открыт экран детали с этим уведомлением.

**Acceptance Scenarios**:

1. **Given** приложение закрыто, **When** пользователь нажимает push с `data.deeplink`
   `osetrovich://notifications/{id}` и существующим уведомлением на сервере, **Then** приложение
   запускается и открывает экран детали этого уведомления.
2. **Given** приложение в фоне, **When** пользователь нажимает push с `data.notification_id`
   `{id}` (без `deeplink` или с пустым `deeplink`), **Then** приложение выходит на передний план и
   открывает экран детали уведомления `{id}`.
3. **Given** пользователь нажимает push, **When** уведомление успешно открыто, **Then** навигация
   выполняется только по `data.deeplink` или `data.notification_id`; текст `notification.body` и
   `notification.title` MUST NOT использоваться для определения целевого экрана или идентификатора.
4. **Given** push содержит `data.deeplink` и `data.notification_id`, **When** приложение
   обрабатывает нажатие, **Then** приоритет у `data.deeplink`; идентификатор в URL MUST совпадать
   с `notification_id` (при расхождении — использовать `deeplink`).
5. **Given** пользователь нажимает push и открывается деталь непрочитанного уведомления,
   **When** экран детали отображается, **Then** уведомление помечается прочитанным и счётчик на
   колокольчике уменьшается (согласно существующему поведению раздела «Уведомления»).

### User Story 3 — In-app баннер при push в foreground (Priority: P2)

Покупатель пользуется приложением, когда приходит push. Помимо обновления счётчика, он MAY увидеть
краткий in-app баннер с заголовком и текстом из `notification.title` и `notification.body`. Нажатие
на баннер открывает тот же экран детали, что и нажатие на системное уведомление.

**Why this priority**: улучшает заметность нового сообщения в foreground; не блокирует основной
сценарий обновления счётчика и навигации по диплинку.

**Independent Test**: при открытом приложении отправить push, убедиться что появился in-app баннер
(если реализован) с корректным заголовком и текстом; нажать на баннер и проверить переход в
деталь уведомления.

**Acceptance Scenarios**:

1. **Given** приложение на переднем плане, **When** приходит push с `notification.title` и
   `notification.body`, **Then** пользователь MAY увидеть in-app баннер с этими текстами.
2. **Given** отображён in-app баннер, **When** пользователь нажимает на него, **Then** открывается
   экран детали уведомления по `data.deeplink` или `data.notification_id` (тот же маршрут, что при
   tap на системное уведомление).
3. **Given** in-app баннер показан, **When** пользователь закрывает баннер без нажатия,
   **Then** счётчик на колокольчике остаётся обновлённым; уведомление остаётся непрочитанным до
   явного открытия детали или «Отметить все прочитанным».

### User Story 4 — Уведомление не найдено (Priority: P2)

Покупатель нажимает push со ссылкой на уведомление, которое уже удалено на сервере или недоступно
этому покупателю. Вместо пустого экрана или сбоя приложение показывает понятное сообщение
«Уведомление не найдено».

**Why this priority**: защищает от устаревших push и race condition между доставкой и удалением
записи; соответствует контракту API.

**Independent Test**: отправить push с `notification_id`, для которого `GET /v1/notifications/{id}`
возвращает 404; нажать на push и убедиться в отображении сообщения «Уведомление не найдено».

**Acceptance Scenarios**:

1. **Given** push содержит `data.notification_id` несуществующего уведомления, **When**
   пользователь нажимает на push и приложение запрашивает детали, **Then** отображается экран или
   состояние с текстом «Уведомление не найдено» на русском языке.
2. **Given** `GET /v1/notifications/{id}` вернул 404, **When** ошибка обработана, **Then**
   приложение не падает; пользователь MAY вернуться к списку уведомлений или на «Главную».
3. **Given** уведомление было удалено после доставки push, **When** пользователь открывает деталь
   из списка, **Then** применяется то же сообщение «Уведомление не найдено».

---

### Edge Cases

- Push приходит без `data.deeplink` и без `data.notification_id` — при нажатии открывается список
  уведомлений (fallback), счётчик в foreground всё равно обновляется.
- Push содержит только `notification` без блока `data` — foreground-обработка обновляет счётчик;
  tap ведёт в список уведомлений.
- Несколько push подряд в foreground — счётчик остаётся согласованным с сервером, не накапливает
  локальные «+1» без синхронизации.
- Пользователь уже на экране детали того же уведомления — повторный tap на push не ломает навигацию.
- `notification_id` в `data` не совпадает с id в `deeplink` — навигация по `deeplink`.
- Невалидный формат `deeplink` (не `osetrovich://notifications/{id}`) — fallback на список
  уведомлений или безопасный экран «Главная» согласно существующим правилам диплинков.
- Сеть недоступна при обновлении счётчика после foreground push — повторная попытка при восстановлении
  связи или при следующем открытии «Главной».
- Дубликат push с тем же `notification_id` — счётчик не увеличивается повторно сверх значения сервера.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: Приложение MUST обрабатывать входящие push-сообщения, когда оно на переднем плане
  (foreground), на Android и iOS — не полагаясь только на системный баннер ОС.
- **FR-002**: При получении push в foreground приложение MUST обновить счётчик непрочитанных на
  колокольчике, синхронизируя его с сервером (через существующий механизм unread-count и/или
  перезагрузку списка уведомлений).
- **FR-003**: При нажатии на push (приложение в фоне или закрыто) приложение MUST прочитать
  `data.deeplink` и/или `data.notification_id` и открыть экран детали соответствующего уведомления.
- **FR-004**: Значение `data.notification_id` MUST интерпретироваться как десятичная строка,
  соответствующая `id` в `GET /v1/notifications/{id}` для текущего покупателя.
- **FR-005**: Значение `data.deeplink` для push об уведомлении MUST иметь формат
  `osetrovich://notifications/{id}`, где `{id}` совпадает с `notification_id`.
- **FR-006**: Приложение MUST NOT использовать `notification.title`, `notification.body` или их
  парсинг для навигации к экрану или определения идентификатора уведомления.
- **FR-007**: Если `GET /v1/notifications/{id}` возвращает 404, приложение MUST показать
  пользователю сообщение «Уведомление не найдено».
- **FR-008**: В foreground приложение MAY показать in-app баннер с `notification.title` и
  `notification.body`; при нажатии на баннер MUST выполняться та же навигация, что и при tap на
  системное уведомление (FR-003).
- **FR-009**: Если в `data` отсутствуют и `deeplink`, и `notification_id`, при нажатии на push
  приложение MUST открыть список уведомлений.
- **FR-010**: Обработка push MUST сохранять обратную совместимость с legacy payload
  (`type` / `targetId`, сырой URL `osetrovich://`) согласно уже принятой в проекте схеме
  диплинков — без изменения приоритетов разрешения маршрута.
- **FR-011**: Все пользовательские тексты ошибок и баннеров MUST быть на русском языке
  (конституция, принцип II).

### Key Entities

- **Push-сообщение (FCM)**: доставляемое сервером сообщение с блоком `notification` (заголовок и
  текст для отображения) и блоком `data` (строковые поля `deeplink`, `notification_id` для
  навигации).
- **In-app уведомление**: запись в разделе «Уведомления» с идентификатором `id`, заголовком, телом,
  датой и статусом прочитанности; источник истины — API покупателя.
- **Счётчик непрочитанных**: число непрочитанных in-app уведомлений, отображаемое на колокольчике
  «Главной».

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: В 95% тестовых отправок push при открытом приложении счётчик на колокольчике
  обновляется в течение 5 секунд после доставки сообщения и совпадает с актуальным количеством
  непрочитанных на сервере.
- **SC-002**: В 100% тестовых сценариев нажатия на push с валидным `data.deeplink` или
  `data.notification_id` пользователь попадает на экран детали указанного уведомления не более чем
  за 3 секунды после tap.
- **SC-003**: В 100% сценариев с ответом API 404 на запрос детали уведомления пользователь видит
  текст «Уведомление не найдено» без аварийного завершения приложения.
- **SC-004**: Ни в одном acceptance-тесте навигация по push не использует парсинг `notification.body`
  или `notification.title` для определения целевого экрана.
- **SC-005**: Покупатель, получивший push в foreground, может увидеть новое непрочитанное
  уведомление в списке без перезапуска приложения в 100% успешных доставок на тестовых устройствах.

## Assumptions

- Регистрация FCM-токена, in-app список уведомлений, колокольчик и экран детали уже реализованы
  (фичи 002, 011); данная фича дорабатывает обработку входящих push и навигацию по новому формату
  `data`.
- Схема диплинков `osetrovich://…` и разрешение маршрутов определены в фиче 016; push использует
  путь `osetrovich://notifications/{id}`.
- Сервер отправляет push в формате FCM с полями `notification` и `data`, как описано во входном
  запросе; все значения в `data` — строки.
- Для обновления счётчика в foreground достаточно запроса актуального unread-count и/или списка
  с сервера, а не локального инкремента.
- Legacy push без `deeplink` / `notification_id` продолжают открывать список уведомлений.
- Авторизация (JWT) требуется для просмотра детали уведомления; неавторизованный пользователь при
  tap на push проходит существующий сценарий входа или видит приглашение авторизоваться.

## Out of Scope

- Изменение серверной логики отправки push и состава рассылок.
- Парсинг номера заказа, типа уведомления или иных сущностей из текста `notification.body`.
- Выбор канала доставки push vs SMS (ответственность сервера).
- Настройка текста push на стороне клиента (отображаются `title` / `body` с сервера).
- Поддержка диплинков на сущности, отличные от уведомлений, в рамках **обязательных** полей order
  push — такие сценарии MAY обрабатываться общим механизмом диплинков, но не являются предметом
  данной фичи.

## Dependencies

- Существующий раздел «Уведомления» и API: `GET /v1/notifications`, `GET /v1/notifications/{id}`,
  `GET /v1/notifications/unread-count`, `POST /v1/notifications/{id}/read`.
- Существующая интеграция push (FCM / AppMetrica Push) и регистрация токена устройства.
- Механизм диплинков `osetrovich://` (фича 016).
