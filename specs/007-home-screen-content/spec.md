## User Scenarios & Testing *(mandatory)*

### User Story 1 — Баннеры с API и переходами по ссылкам (Priority: P1)

Покупатель на вкладке «Главная» видит карусель баннеров с реальными изображениями,
загруженными с сервера. Нажатие на баннер открывает целевой контент: внешний сайт в
браузере устройства, детальную страницу акции, новости или товара внутри приложения.
Поведение карусели (автопрокрутка, свайп, отступ от шапки) сохраняется из предыдущей
реализации главной.

**Why this priority**: баннеры — первый контент на главной; без реальных данных и рабочих
ссылок промо-канал не приносит ценности.

**Independent Test**: открыть «Главную», убедиться, что баннеры загружаются с изображениями
из API; нажать баннеры с разными типами ссылок и проверить корректный переход.

**Acceptance Scenarios**:

1. **Given** пользователь на вкладке «Главная», **When** данные баннеров успешно загружены,
   **Then** в карусели отображаются баннеры с фотографиями из ответа сервера (не
   placeholder-текст «Баннер N»).
2. **Given** баннер содержит ссылку на внешний ресурс, **When** пользователь нажимает на
   баннер, **Then** открывается внешняя ссылка во встроенном или системном браузере
   устройства.
3. **Given** баннер содержит ссылку на акцию, **When** пользователь нажимает на баннер,
   **Then** открывается детальная страница соответствующей акции внутри приложения.
4. **Given** баннер содержит ссылку на новость, **When** пользователь нажимает на баннер,
   **Then** открывается детальная страница соответствующей новости внутри приложения.
5. **Given** баннер содержит ссылку на товар, **When** пользователь нажимает на баннер,
   **Then** открывается страница товара внутри приложения.
6. **Given** баннер не содержит ссылки, **When** пользователь нажимает на баннер,
   **Then** никакой навигации не происходит; карусель продолжает работать штатно.
7. **Given** сервер вернул меньше трёх баннеров, **When** карусель отображается,
   **Then** показывается столько баннеров, сколько вернул источник; карусель работает при
   наличии хотя бы одного баннера.

### User Story 2 — Горизонтальная лента «Товары недели» (Priority: P1)

Покупатель на «Главной» видит блок «Товары недели» — горизонтально прокручиваемую ленту
товаров, загруженных с сервера. Каждая карточка показывает ключевую информацию о товаре;
нажатие открывает страницу товара; с карточки можно добавить товар в корзину по тому же
принципу, что в каталоге.

**Why this priority**: лента подсвечивает актуальные предложения и стимулирует покупку без
перехода в каталог.

**Independent Test**: открыть «Главную», проверить заголовок «Товары недели», горизонтальный
скролл, переход на товар и добавление в корзину с обновлением бейджа корзины.

**Acceptance Scenarios**:

1. **Given** пользователь на «Главной», **When** данные «Товаров недели» успешно загружены,
   **Then** отображается заголовок «Товары недели» и горизонтальная лента карточек товаров.
2. **Given** лента содержит несколько товаров, **When** пользователь свайпает влево/вправо,
   **Then** видны следующие карточки без перезагрузки экрана.
3. **Given** карточка товара в ленте, **When** пользователь нажимает на карточку (кроме
   зоны управления количеством), **Then** открывается страница этого товара.
4. **Given** товар ещё не в корзине, **When** пользователь нажимает кнопку добавления на
   карточке в ленте, **Then** товар добавляется в корзину; бейдж на вкладке «Корзина»
   обновляется реактивно.
5. **Given** сервер вернул пустой список «Товаров недели», **When** пользователь смотрит на
   «Главную», **Then** блок «Товары недели» не отображается (или скрыт без пустого
   заголовка).

### User Story 3 — Блок текущего заказа на «Главной» (Priority: P1)

Авторизованный покупатель с активным или недавно завершённым заказом видит на «Главной»
блок «История заказов» с информацией о текущем заказе: статус, состав (товары с количеством
и ценой), итоговая сумма и кнопка «Связаться с оператором».

**Why this priority**: статус заказа на главной снижает тревожность покупателя и даёт быстрый
доступ к поддержке.

**Independent Test**: авторизоваться под пользователем с активным заказом, открыть
«Главную» и проверить отображение статуса, позиций, суммы и кнопки связи.

**Acceptance Scenarios**:

1. **Given** пользователь авторизован и у него есть текущий заказ, **When** он открывает
   «Главную», **Then** отображается блок «История заказов» с актуальным статусом заказа
   из списка: «Принят», «В обработке», «Сборка», «Доставка» или «Выполнено».
2. **Given** отображается текущий заказ, **When** пользователь смотрит на состав,
   **Then** виден список товаров заказа: название, количество и цена по каждой позиции.
3. **Given** отображается текущий заказ, **When** пользователь смотрит на итог,
   **Then** отображается итоговая сумма заказа.
4. **Given** отображается текущий заказ в любом статусе, **When** пользователь нажимает
   «Связаться с оператором», **Then** инициируется звонок на номер поддержки магазина
   (+7 812 564-55-48).
5. **Given** пользователь не авторизован, **When** он открывает «Главную»,
   **Then** блок «История заказов» не отображается.
6. **Given** пользователь авторизован, но текущего заказа нет, **When** он открывает
   «Главную», **Then** блок «История заказов» не отображается.
7. **Given** пользователь вышел из аккаунта и вошёл под другим (или тем же) номером,
   **When** он возвращается на «Главную», **Then** блок «История заказов» отображает данные
   актуального пользователя: скрыт при отсутствии заказа или обновлён при наличии заказа
   без ручного обновления экрана.

### User Story 4 — Оценка выполненного заказа (Priority: P2)

Авторизованный покупатель с заказом в статусе «Выполнено», по которому ещё не принято
решение об оценке, видит приглашение оценить заказ с кнопками «Оценить» и «Пропустить».

**Why this priority**: обратная связь улучшает сервис; сценарий актуален только после
завершения доставки.

**Independent Test**: открыть «Главную» с выполненным неоценённым заказом; проверить текст
приглашения и обе кнопки; убедиться, что после оценки или пропуска приглашение исчезает.

**Acceptance Scenarios**:

1. **Given** текущий заказ в статусе «Выполнено» и оценка ещё не дана и не пропущена,
   **When** пользователь открывает «Главную», **Then** в блоке заказа отображается текст
   «Пожалуйста, оцените ваш заказ, это поможет нам стать лучше!», кнопки «Оценить» и
   «Пропустить».
2. **Given** отображается приглашение к оценке, **When** пользователь нажимает «Оценить»,
   **Then** открывается интерфейс оценки заказа; после успешной отправки оценки приглашение
   скрывается, статус заказа и состав остаются видимыми.
3. **Given** отображается приглашение к оценке, **When** пользователь нажимает «Пропустить»,
   **Then** приглашение скрывается без отправки оценки; блок заказа остаётся на экране.
4. **Given** текущий заказ не в статусе «Выполнено», **When** пользователь открывает
   «Главную», **Then** приглашение к оценке не отображается.

### User Story 5 — Повтор заказа (Priority: P2)

После оценки или пропуска оценки выполненного заказа покупатель может одним нажатием
повторить заказ: все товары из заказа попадают в корзину, после чего открывается вкладка
«Корзина».

**Why this priority**: ускоряет повторные покупки; зависит от завершённого заказа и решения
по оценке.

**Independent Test**: для выполненного заказа с пропущенной оценкой нажать «Повторить заказ»
при пустой и непустой корзине; проверить состав корзины и переход на вкладку «Корзина».

**Acceptance Scenarios**:

1. **Given** текущий заказ в статусе «Выполнено» и оценка уже дана или пропущена,
   **When** пользователь открывает «Главную», **Then** в блоке заказа отображается кнопка
   «Повторить заказ» (приглашение к оценке не показывается).
2. **Given** корзина пуста, **When** пользователь нажимает «Повторить заказ»,
   **Then** все позиции текущего заказа с исходными количествами добавляются в корзину;
   пользователь перенаправляется на вкладку «Корзина».
3. **Given** корзина уже содержит товары, **When** пользователь нажимает «Повторить заказ»,
   **Then** позиции заказа **добавляются** к существующим (одинаковые товары суммируют
   количество); пользователь перенаправляется на вкладку «Корзина».
4. **Given** один или несколько товаров из заказа больше недоступны в каталоге,
   **When** пользователь нажимает «Повторить заказ», **Then** в корзину попадают только
   доступные позиции; пользователь видит понятное сообщение на русском языке о пропущенных
   товарах; переход на «Корзину» выполняется, если хотя бы одна позиция добавлена.
5. **Given** ни один товар из заказа недоступен, **When** пользователь нажимает
   «Повторить заказ», **Then** корзина не меняется; отображается сообщение, что повторить
   заказ невозможно.

### Edge Cases

- Что происходит при ошибке загрузки баннеров? — Карусель скрывается или показывается
  состояние ошибки с кнопкой «Повторить»; остальные блоки «Главной» продолжают работать.
- Что происходит при ошибке загрузки «Товаров недели»? — Блок ленты скрывается или показывает
  ошибку с повтором; баннеры и другие блоки не блокируются.
- Что происходит при ошибке загрузки текущего заказа? — Блок «История заказов» скрывается или
  показывает ошибку с повтором; остальной контент главной доступен.
- Что если изображение баннера не загрузилось? — Показывается нейтральная заглушка; нажатие
  по ссылке (если есть) остаётся рабочим.
- Что если целевой объект внутренней ссылки удалён (акция, новость, товар)? — Пользователь
  видит понятное сообщение на русском языке; приложение не падает.
- Что если заказ в статусе «Выполнено», но пользователь ещё не оценил и не пропустил? —
  Кнопка «Повторить заказ» **не** отображается; видны только «Оценить» и «Пропустить».
- Что происходит после «Повторить заказ»? — Блок текущего заказа на «Главной» может
  оставаться до появления нового заказа; повторное нажатие снова добавляет позиции в корзину.
- Что если пользователь разлогинился на «Главной»? — Блок заказа исчезает; баннер
  «Авторизуйтесь» отображается для гостя.
- Что если пользователь перелогинился (вышел и вошёл снова или сменил аккаунт)? — Блок
  «История заказов» MUST перезапросить текущий заказ по новой сессии и отобразить актуальные
  данные; виджет блока пересоздаётся при смене заказа.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: Карусель баннеров на «Главной» MUST загружать список баннеров с сервера при
  открытии экрана (на этапе разработки — мок-данные по контракту).
- **FR-002**: Каждый баннер MUST отображать фотографию из данных сервера.
- **FR-003**: Каждый баннер MAY содержать ссылку одного из типов: внешний URL, акция,
  новость, товар; тип и идентификатор цели MUST приходить в ответе API.
- **FR-004**: Нажатие на баннер с внешней ссылкой MUST открывать URL во внешнем или
  встроенном браузере устройства.
- **FR-005**: Нажатие на баннер со ссылкой на акцию, новость или товар MUST открывать
  соответствующий экран внутри приложения.
- **FR-006**: Поведение карусели (автопрокрутка каждые 5 секунд, ручной свайп, отступ от
  шапки, peek соседних баннеров при нескольких элементах) MUST сохраняться согласно
  реализации из `002-notifications-home`.
- **FR-007**: На «Главной» MUST отображаться блок «Товары недели» с горизонтальной лентой
  товаров, загружаемой с сервера.
- **FR-008**: Карточка товара в ленте MUST показывать фото, название, вес и цену; MUST
  поддерживать добавление в корзину и переход на страницу товара по тем же правилам, что
  карточка в каталоге (`004-product-catalog`).
- **FR-009**: Блок «Товары недели» MUST скрываться, если сервер вернул пустой список.
- **FR-010**: Для авторизованного пользователя MUST запрашиваться текущий заказ с сервера.
- **FR-011**: Блок «История заказов» MUST отображаться только для авторизованного
  пользователя при наличии текущего заказа.
- **FR-012**: Блок заказа MUST показывать статус из фиксированного набора: «Принят»,
  «В обработке», «Сборка», «Доставка», «Выполнено».
- **FR-013**: Блок заказа MUST показывать состав: список товаров с количеством и ценой по
  позиции, а также итоговую сумму заказа.
- **FR-014**: Блок заказа MUST содержать кнопку «Связаться с оператором», инициирующую
  звонок на +7 812 564-55-48.
- **FR-015**: Для заказа в статусе «Выполнено» без принятого решения об оценке MUST
  отображаться текст «Пожалуйста, оцените ваш заказ, это поможет нам стать лучше!» и кнопки
  «Оценить» и «Пропустить».
- **FR-016**: Нажатие «Оценить» MUST открывать сценарий оценки заказа и сохранять результат
  на сервере; после успеха приглашение к оценке MUST скрываться.
- **FR-017**: Нажатие «Пропустить» MUST фиксировать отказ от оценки без отправки оценки;
  приглашение MUST скрываться.
- **FR-018**: Для заказа в статусе «Выполнено» после оценки или пропуска MUST отображаться
  кнопка «Повторить заказ».
- **FR-019**: «Повторить заказ» MUST добавить все доступные позиции заказа в корзину с
  исходными количествами; при непустой корзине позиции MUST объединяться с существующими.
- **FR-020**: После успешного «Повторить заказ» пользователь MUST быть перенаправлен на
  вкладку «Корзина»; бейдж корзины MUST обновиться реактивно.
- **FR-021**: Все пользовательские тексты, кнопки и сообщения в scope фичи MUST быть на
  русском языке.
- **FR-022**: Блок «Связаться» (`HomeContactButton`) и баннер «Авторизуйтесь» для гостей
  MUST сохраняться на «Главной» согласно предыдущим фичам.
- **FR-023**: При смене сессии авторизации (вход, выход, смена пользователя) блок
  «История заказов» MUST реактивно обновляться: запрос текущего заказа привязан к идентичности
  сессии, а не только к флагу «авторизован».

### Key Entities

- **Баннер главной**: идентификатор, URL изображения, порядок сортировки, тип ссылки
  (внешняя / акция / новость / товар / отсутствует), целевой идентификатор или URL.
- **Товар недели**: те же ключевые атрибуты, что у товара каталога для отображения в
  карточке (идентификатор, фото, название, вес, цена); порядок в ленте.
- **Текущий заказ**: идентификатор, номер, статус (из пяти значений), список позиций
  (товар, количество, цена), итоговая сумма, флаги «оценка отправлена» и «оценка пропущена».
- **Оценка заказа**: привязка к заказу, выбранная пользователем оценка (шкала определяется
  на этапе планирования), дата отправки.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: Пользователь видит баннеры с реальными изображениями на «Главной» в течение
  3 секунд после открытия экрана при стабильном соединении.
- **SC-002**: 100% баннеров с валидной ссылкой открывают ожидаемый целевой экран или URL
  при нажатии.
- **SC-003**: Пользователь может просмотреть всю ленту «Товары недели» горизонтальным
  свайпом без перезагрузки «Главной».
- **SC-004**: Добавление товара из ленты «Товары недели» обновляет бейдж корзины в течение
  1 секунды без ручного обновления экрана.
- **SC-005**: Авторизованный пользователь с активным заказом видит актуальный статус и
  состав заказа на «Главной» без перехода в профиль.
- **SC-006**: Сценарий «Повторить заказ» (пустая корзина) завершается переходом на
  «Корзину» с полным составом заказа менее чем за 5 секунд.
- **SC-007**: 100% видимых пользовательских строк интерфейса в scope фичи — на русском

## Assumptions

- «Текущий заказ» — последний заказ пользователя, который ещё не архивирован: активный
  (статусы до «Выполнено» включительно) или выполненный, ожидающий оценки/повтора; полный
  список истории заказов вне scope этой фичи.
- Данные баннеров, «Товаров недели» и текущего заказа загружаются с REST API; на этапе
  разработки используются моки по OpenAPI (конституция, принцип VI).
- Порядок блоков на «Главной» сверху вниз: шапка → баннеры → «Связаться» → «Товары недели»
  → «История заказов» → баннер «Авторизуйтесь» (для гостей). Точный порядок MAY быть
  уточнён в plan.md без изменения функциональных требований.
- Интерфейс оценки заказа (шкала, комментарий) детализируется в plan.md; минимум —
  возможность выбрать оценку и подтвердить отправку.
- Повтор заказа использует актуальные цены товаров из каталога на момент добавления в
  корзину, а не зафиксированные в заказе.
- Блок заказа не показывается неавторизованным пользователям; для просмотра заказов в
  профиле действуют отдельные фичи (`003-user-profile`, `005-cart-checkout`).
- Существующие экраны акции, новости и товара переиспользуются для внутренних переходов
  с баннеров (`004-product-catalog`, `006-promotions-news`).
- В мок-API для ручной проверки заказов на «Главной» заданы демо-номера: `+79001111111`
  (активная доставка), `+79002222222` (выполнен, оценка не дана), `+79003333333`
  (выполнен, оценка пропущена — доступен «Повторить заказ»); профиль и заказы в моке
  привязаны к номеру телефона пользователя.
